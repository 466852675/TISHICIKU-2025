# 多智能体架构模式

多智能体架构将工作分布在多个语言模型实例上，每个实例都有自己的上下文窗口。当设计良好时，这种分布使能力超越单智能体限制。当设计不佳时，它会引入抵消收益的协调开销。关键洞察是，子智能体主要存在于隔离上下文，而不是拟人化角色分工。

## 何时激活

在以下情况下激活此技能：
- 单智能体上下文限制约束任务复杂性
- 任务自然分解为并行子任务
- 不同的子任务需要不同的工具集或系统提示
- 构建必须同时处理多个领域的系统
- 将智能体能力扩展到超出单上下文限制
- 设计具有多个专门组件的生产智能体系统

## 核心概念

多智能体系统通过分布解决单智能体上下文限制。三种主导模式存在：监督者/编排器用于集中控制，点对点/群体用于灵活交接，分层用于分层抽象。关键设计原则是上下文隔离——子智能体主要存在于分区上下文而不是模拟组织角色。

有效的多智能体系统需要显式协调协议、避免谄媚的共识机制，以及对故障模式（包括瓶颈、分歧和错误传播）的仔细关注。

## 详细主题

### 为什么使用多智能体架构

**上下文瓶颈**
单智能体在推理能力、上下文管理和工具协调方面面临固有限制。随着任务变得更复杂，上下文窗口被累积的历史、检索的文档和工具输出填满。性能根据可预测的模式下降：中间丢失效应、注意力稀缺和上下文污染。

多智能体架构通过将工作分区到多个上下文窗口来解决这些限制。每个智能体在专注于其子任务的干净上下文中操作。结果在协调层聚合，没有任何单个上下文承担全部负担。

**令牌经济学现实**
多智能体系统比单智能体方法消耗显著更多的令牌。生产数据显示：

| 架构 | 令牌倍数 | 用例 |
|--------------|------------------|----------|
| 单智能体聊天 | 1× 基线 | 简单查询 |
| 带工具的单智能体 | ~4× 基线 | 使用工具的任务 |
| 多智能体系统 | ~15× 基线 | 复杂研究/协调 |

对 BrowseComp 评估的研究发现，三个因素解释了 95% 的性能差异：令牌使用（80% 的方差）、工具调用次数和模型选择。这验证了将工作分布在具有单独上下文窗口的智能体上以添加并行推理能力的多智能体方法。

关键的是，升级到更好的模型通常比加倍令牌预算提供更大的性能收益。Claude Sonnet 4.5 显示出比在早期 Sonnet 版本上加倍令牌更大的收益。GPT-5.2 的思考模式同样优于原始令牌增加。这表明模型选择和多智能体架构是互补策略。

**并行化论证**
许多任务包含单智能体必须顺序执行的可并行子任务。研究任务可能需要搜索多个独立来源、分析不同文档或比较竞争方法。单智能体顺序处理这些，每一步都累积上下文。

多智能体架构将每个子任务分配给具有新上下文的专用智能体。所有智能体同时工作，然后将结果返回给协调器。总实际时间接近最长子任务的持续时间，而不是所有子任务的总和。

**专业化论证**
不同的任务受益于不同的智能体配置：不同的系统提示、不同的工具集、不同的上下文结构。通用智能体必须在上下文中携带所有可能的配置。专门智能体只携带它们需要的东西。

多智能体架构使专业化而不会产生组合爆炸。协调器路由到专门智能体；每个智能体在其领域优化的精简上下文中操作。

### 架构模式

**模式 1：监督者/编排器**
监督者模式将中央智能体置于控制之下，委派给专家并综合结果。监督者维护全局状态和轨迹，将用户目标分解为子任务，并路由到适当的工作者。

```
用户查询 -> 监督者 -> [专家, 专家, 专家] -> 聚合 -> 最终输出
```

何时使用：具有清晰分解的复杂任务、需要跨领域协调的任务、人类监督很重要的任务。

优点：对工作流的严格控制、更容易实现人在回路干预、确保遵守预定义计划。

缺点：监督者上下文成为瓶颈、监督者故障级联到所有工作者、"传话游戏"问题，监督者错误地解释子智能体响应。

**传话游戏问题和解决方案**
LangGraph 基准测试发现，由于"传话游戏"问题，监督者架构最初比优化版本表现差 50%，监督者错误地解释子智能体响应，丢失保真度。

修复：实现 `forward_message` 工具允许子智能体将响应直接传递给用户：

```python
def forward_message(message: str, to_user: bool = True):
    """
    将子智能体响应直接转发给用户，无需监督者综合。
    
    使用时机：
    - 子智能体响应是最终和完整的
    - 监督者综合会丢失重要细节
    - 响应格式必须完全保留
    """
    if to_user:
        return {"type": "direct_response", "content": message}
    return {"type": "supervisor_input", "content": message}
```

使用此模式，群体架构略微优于监督者，因为子智能体直接响应用户，消除翻译错误。

实现说明：实现直接传递机制，允许子智能体在适当时将响应直接传递给用户，而不是通过监督者综合。

**模式 2：点对点/群体**
点对点模式移除中央控制，允许智能体基于预定义协议直接通信。任何智能体都可以通过显式交接机制将控制权转移给任何其他智能体。

```python
def transfer_to_agent_b():
    return agent_b  # 通过函数返回交接

agent_a = Agent(
    name="智能体 A",
    functions=[transfer_to_agent_b]
)
```

何时使用：需要灵活探索的任务、严格计划适得其反的任务、具有无法预先分解的涌现需求的任务。

优点：没有单点故障、对广度优先探索有效扩展、使涌现问题解决行为成为可能。

缺点：随着智能体数量增加协调复杂性、没有中央状态保持器时存在分歧风险、需要健壮的收敛约束。

实现说明：定义具有状态传递的显式交接协议。确保智能体可以将它们的上下文需求传达给接收智能体。

**模式 3：分层**
分层结构将智能体组织成抽象层：战略层、规划层和执行层。战略层智能体定义目标和约束；规划层智能体将目标分解为可操作的计划；执行层智能体执行原子任务。

```
战略层（目标定义） -> 规划层（任务分解） -> 执行层（原子任务）
```

何时使用：具有清晰分层结构的大型项目、具有管理层的企业工作流、需要高层规划和详细执行的任务。

优点：镜像组织结构、关注点清晰分离、在不同级别启用不同的上下文结构。

缺点：层之间的协调开销、战略和执行之间潜在的不一致、复杂的错误传播。

### 上下文隔离作为设计原则

多智能体架构的主要目的是上下文隔离。每个子智能体在专注于其子任务的干净上下文窗口中操作，而不携带来自其他子任务的累积上下文。

**隔离机制**
完整上下文委托：对于子智能体需要完整理解的复杂任务，规划者共享其整个上下文。子智能体有自己的工具和指令，但接收其决策的完整上下文。

指令传递：对于简单、定义良好的子任务，规划者通过函数调用创建指令。子智能体仅接收其特定任务所需的指令。

文件系统内存：对于需要共享状态的复杂任务，智能体读取和写入持久存储。文件系统充当协调机制，避免来自共享状态传递的上下文膨胀。

**隔离权衡**
完整上下文委托提供最大能力但违背了子智能体的目的。指令传递保持隔离但限制子智能体灵活性。文件系统内存使共享状态无需上下文传递，但引入延迟和一致性挑战。

正确的选择取决于任务复杂性、协调需求和可接受的延迟。

### 共识和协调

**投票问题**
简单多数投票将弱模型的幻觉视为等同于强模型的推理。没有干预，多智能体讨论由于固有的达成一致偏见而沦为对错误前提的共识。

**加权投票**
按信心或专业知识加权智能体投票。具有更高信心或领域专业知识的智能体在最终决策中携带更多权重。

**辩论协议**
辩论协议要求智能体在多轮中批评彼此的输出。对抗性批评通常在复杂推理上产生比协作共识更高的准确性。

**基于触发的干预**
监控多智能体交互以获取特定行为标记。当讨论没有进展时激活停滞触发器。当智能体在没有独特推理的情况下模仿彼此的答案时检测到谄媚触发器。

### 框架考虑

不同的框架以不同的理念实现这些模式。LangGraph 使用基于图的状态机和显式节点与边。AutoGen 使用对话/事件驱动模式和 GroupChat。CrewAI 使用基于角色的流程和分层团队结构。

## 实践指导

### 故障模式和缓解

**故障：监督者瓶颈**
监督者从所有工作者累积上下文，容易饱和和退化。

缓解：实现输出模式约束，使工作者仅返回提炼的摘要。使用检查点在不携带完整历史的情况下持久化监督者状态。

**故障：协调开销**
智能体通信消耗令牌并引入延迟。复杂的协调可能抵消并行化收益。

缓解：通过清晰的交接协议最小化通信。尽可能批处理结果。使用异步通信模式。

**故障：分歧**
在没有中央协调的情况下追求不同目标的智能体可能偏离预期目标。

缓解：为每个智能体定义清晰的目标边界。实现收敛检查以验证向共享目标的进展。对智能体执行使用生存时间限制。

**故障：错误传播**
一个智能体输出中的错误传播到消费该输出的下游智能体。

缓解：在传递给消费者之前验证智能体输出。使用断路器实现重试逻辑。尽可能使用幂等操作。

## 示例

**示例 1：研究团队架构**
```text
监督者
├── 研究员（网络搜索、文档检索）
├── 分析员（数据分析、统计）
├── 事实核查员（验证、确认）
└── 撰写员（报告生成、格式化）
```

**示例 2：交接协议**
```python
def handle_customer_request(request):
    if request.type == "billing":
        return transfer_to(billing_agent)
    elif request.type == "technical":
        return transfer_to(technical_agent)
    elif request.type == "sales":
        return transfer_to(sales_agent)
    else:
        return handle_general(request)
```

## 指南

1. 将上下文隔离设计为多智能体系统的主要优势
2. 基于协调需求而非组织隐喻选择架构模式
3. 实现具有状态传递的显式交接协议
4. 使用加权投票或辩论协议达成共识
5. 监控监督者瓶颈并实现检查点
6. 在智能体之间传递之前验证输出
7. 设置生存时间限制以防止无限循环
8. 显式测试故障场景

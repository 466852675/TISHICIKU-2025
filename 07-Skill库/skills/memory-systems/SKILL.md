# 记忆系统设计

记忆提供持久层，使代理能够跨会话保持连续性并对累积知识进行推理。简单代理完全依赖上下文作为记忆，会话结束时丢失所有状态。复杂的代理实现分层记忆架构，平衡即时上下文需求与长期知识保留。从向量存储到知识图谱再到时序知识图谱的演进代表了对结构化记忆的日益投资，以改善检索和推理。

## 何时激活

在以下情况下激活此技能：
- 构建必须跨会话持久化的代理
- 需要跨对话保持实体一致性
- 对累积知识实施推理
- 设计从过去交互中学习的系统
- 创建随时间增长的知识库
- 构建跟踪状态变化的时序感知系统

## 核心概念

记忆存在于从即时上下文到永久存储的频谱上。在一个极端，上下文窗口中的工作记忆提供零延迟访问，但会话结束时消失。在另一个极端，永久存储无限期持久化，但需要检索才能进入上下文。

简单向量存储缺乏关系和时序结构。知识图谱保留关系以进行推理。时序知识图谱添加有效期以进行时序感知查询。实施选择取决于查询复杂性、基础设施约束和准确性要求。

## 详细主题

### 记忆架构基础

**上下文-记忆频谱**
记忆存在于从即时上下文到永久存储的频谱上。在一个极端，上下文窗口中的工作记忆提供零延迟访问，但会话结束时消失。在另一个极端，永久存储无限期持久化，但需要检索才能进入上下文。有效的架构使用此频谱上的多个层。

该频谱包括工作记忆（上下文窗口、零延迟、易失性）、短期记忆（会话持久、可搜索、易失性）、长期记忆（跨会话持久、结构化、半永久）和永久记忆（归档、可查询、永久）。每层具有不同的延迟、容量和持久性特征。

**为什么简单向量存储不足**
向量RAG通过将查询和文档嵌入共享的嵌入空间来提供语义检索。相似性搜索检索语义最相似的文档。这对文档检索很有效，但缺乏代理记忆的结构。

向量存储丢失关系信息。如果代理了解到"客户X在日期Z购买了产品Y"，向量存储可以直接检索此事实。但它无法回答"购买产品Y的客户还购买了什么产品？"，因为关系结构未被保留。

向量存储也在时序有效性方面遇到困难。事实随时间变化，但向量存储除了显式元数据和过滤外，没有区分"当前事实"和"过时事实"的机制。

**向基于图的记忆迁移**
知识图谱保留实体之间的关系。与孤立的文档块不同，图谱编码实体A具有与实体B的关系R。这支持遍历关系而非仅相似性的查询。

时序知识图谱为事实添加有效期。每个事实都有"有效自"和可选的"有效至"时间戳。这支持在特定时间点重建知识的时序旅行查询。

**基准性能比较**
深度记忆检索（DMR）基准提供了跨记忆架构的具体性能数据：

| 记忆系统 | DMR准确率 | 检索延迟 | 说明 |
|----------|-----------|----------|------|
| Zep (时序KG) | 94.8% | 2.58秒 | 最佳准确率，快速检索 |
| MemGPT | 93.4% | 可变 | 良好的整体性能 |
| GraphRAG | ~75-85% | 可变 | 比基线RAG提升20-35% |
| 向量RAG | ~60-70% | 快速 | 丢失关系结构 |
| 递归摘要 | 35.3% | 低 | 严重的信息丢失 |

Zep相比完整上下文基线展示了90%的检索延迟减少（GPT-5.2为2.58秒 vs 28.9秒）。这种效率来自检索仅相关的子图而非整个上下文历史。

GraphRAG在复杂推理任务中比基线RAG实现约20-35%的准确率提升，并通过基于社区的摘要将幻觉减少高达30%。

### 记忆层架构

**第1层：工作记忆**
工作记忆是上下文窗口本身。它提供对当前正在处理的信息的即时访问，但容量有限且会话结束时消失。

工作记忆使用模式包括代理跟踪中间结果的便签计算、保留当前任务对话的对话历史、跟踪活动目标进展的当前任务状态，以及保存当前使用信息的活跃检索文档。

通过仅保留活跃信息、在信息脱离注意力之前总结已完成的工作，以及将关键信息放在注意力偏好的位置来优化工作记忆。

**第2层：短期记忆**
短期记忆在当前会话中持久化，但不在跨会话中持久化。它提供搜索和检索能力，而没有永久存储的延迟。

常见实现包括会话结束前的会话范围数据库、指定会话目录中的文件系统存储，以及按会话ID键控的内存缓存。

短期记忆用例包括跨轮次跟踪对话状态而不填充上下文、存储以后可能需要的工具调用的中间结果、维护任务清单和进度跟踪，以及在会话内缓存检索到的信息。

**第3层：长期记忆**
长期记忆跨会话无限期持久化。它使代理能够从过去的交互中学习并随时间构建知识。

长期记忆实现范围从简单的键值存储到复杂的图数据库。选择取决于要建模的关系复杂性、所需的查询模式和可接受的基础设施复杂性。

长期记忆用例包括跨会话学习用户偏好、随时间增长的领域知识库、具有关系历史的实体注册表，以及可重用的成功模式存储。

**第4层：实体记忆**
实体记忆专门跟踪关于实体（人、地点、概念、对象）的信息以保持一致性。这创建了一个原始知识图谱，其中实体在多次交互中被识别。

实体记忆通过跟踪一个对话中提到的"John Doe"与另一个对话中是同一个人来维护实体身份。它通过随时间存储发现的关于实体的事实来维护实体属性。它通过在发现时跟踪实体之间的关系来维护实体关系。

**第5层：时序知识图谱**
时序知识图谱用显式有效期扩展实体记忆。事实不仅是真或假，而是在特定时间范围内为真。

这支持诸如"用户X在日期Y的地址是什么？"的查询，通过检索该日期范围有效的事实。它防止过时信息与新数据矛盾时的上下文冲突。它支持关于实体如何随时间变化的时序推理。

### 记忆实施模式

**模式1：文件系统作为记忆**
文件系统本身可以作为记忆层。此模式简单，不需要额外的基础设施，并启用与基于文件的上下文相同的即时加载。

实施使用文件系统层次结构进行组织。使用传达意义的命名约定。以结构化格式（JSON、YAML）存储事实。在文件名或元数据中使用时间戳进行时序跟踪。

优势：简单、透明、可移植。
劣势：无语义搜索、无关系跟踪、需要手动组织。

**模式2：带元数据的向量RAG**
增强的带有丰富元数据的向量存储提供具有过滤能力的语义搜索。

实施嵌入事实或文档并存储带有元数据（包括实体标签、时序有效性、来源归属和置信度分数）。查询包括元数据过滤以及语义搜索。

**模式3：知识图谱**
知识图谱显式建模实体和关系。实施定义实体类型和关系类型，使用图数据库或属性图存储，并维护常见查询模式的索引。

**模式4：时序知识图谱**
时序知识图谱为事实添加有效期，支持时序旅行查询并防止过时信息的上下文冲突。

### 记忆检索模式

**语义检索**
使用嵌入相似性搜索检索与当前查询语义相似的记忆。

**基于实体的检索**
通过遍历图关系检索与特定实体相关的所有记忆。

**时序检索**
使用有效期过滤器检索在特定时间或时间范围内有效的记忆。

### 记忆整合

记忆随时间累积，需要整合以防止无限制增长并删除过时信息。

**整合触发器**
在显著记忆累积后、检索返回过多过时结果时、按计划定期或请求显式整合时触发整合。

**整合过程**
识别过时事实、合并相关事实、更新有效期、归档或删除过时事实、重建索引。

## 实用指导

### 与上下文集成

记忆必须与上下文系统集成才有用。使用即时记忆加载在需要时检索相关记忆。使用策略性注入将记忆放在注意力偏好的位置。

### 记忆系统选择

根据需求选择记忆架构：
- 简单持久化需求：文件系统记忆
- 语义搜索需求：带元数据的向量RAG
- 关系推理需求：知识图谱
- 时序有效性需求：时序知识图谱

## 示例

**示例1：实体跟踪**
```python
# 跨对话跟踪实体
def remember_entity(entity_id, properties):
    memory.store({
        "type": "entity",
        "id": entity_id,
        "properties": properties,
        "last_updated": now()
    })

def get_entity(entity_id):
    return memory.retrieve_entity(entity_id)
```

**示例2：时序查询**
```python
# 用户X在2024年1月15日的地址是什么？
def query_address_at_time(user_id, query_time):
    return temporal_graph.query("""
        MATCH (user)-[r:LIVES_AT]->(address)
        WHERE user.id = $user_id
        AND r.valid_from <= $query_time
        AND (r.valid_until IS NULL OR r.valid_until > $query_time)
        RETURN address
    """, {"user_id": user_id, "query_time": query_time})
```

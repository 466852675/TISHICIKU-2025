# 上下文降级模式

语言模型随着上下文长度增加表现出可预测的降级模式。理解这些模式对于诊断故障和设计弹性系统至关重要。上下文降级不是二元状态，而是以几种不同方式表现的性能降级连续体。

## 何时激活

在以下情况下激活此技能：
- 长对话期间智能体性能意外降级
- 调试智能体产生不正确或无关输出的情况
- 设计必须可靠处理大上下文的系统
- 为生产系统评估上下文工程选择
- 调查智能体输出中的"中间丢失"现象
- 分析智能体行为中的上下文相关故障

## 核心概念

上下文降级通过几种不同的模式表现。中间丢失现象导致上下文中心的信息接收较少注意力。上下文污染发生在错误通过重复引用而复合时。上下文分心发生在无关信息压倒相关内容时。上下文混淆发生在模型无法确定哪个上下文适用时。上下文冲突在累积信息直接冲突时发展。

这些模式是可预测的，可以通过压缩、掩码、分区和隔离等架构模式来缓解。

## 详细主题

### 中间丢失现象

最著名的降级模式是"中间丢失"效应，其中模型表现出 U 形注意力曲线。上下文开头和结尾的信息接收可靠的注意力，而埋在中间的信息遭受显著降低的召回准确性。

**经验证据**
研究表明，放置在上下文中间的相关信息与开头或结尾的相同信息相比，召回准确性降低 10-40%。这不是模型的失败，而是注意力机制和训练数据分布的结果。

模型将大量注意力分配给第一个令牌（通常是 BOS 令牌）以稳定内部状态。这创建了一个吸收注意力预算的"注意力汇"。随着上下文增长，有限的预算被拉伸得更薄，中间令牌无法获得足够可靠的检索注意力权重。

**实际影响**
设计上下文放置时考虑注意力模式。将关键信息放在上下文的开头或结尾。考虑信息将被直接查询还是需要支持推理——如果是后者，放置的重要性较低，但整体信号质量更重要。

对于长文档或对话，使用在受青睐的注意力位置呈现关键信息的摘要结构。使用显式章节标题和转换帮助模型导航结构。

### 上下文污染

上下文污染发生在幻觉、错误或不正确信息进入上下文并通过重复引用而复合时。一旦污染，上下文会创建强化不正确信念的反馈循环。

**污染如何发生**
污染通常通过三种途径进入。首先，工具输出可能包含错误或智能体接受为事实的意外格式。其次，检索的文档可能包含智能体纳入推理的不正确或过时信息。第三，模型生成的摘要或中间输出可能引入在上下文中持续的幻觉。

复合效应是严重的。如果智能体的目标部分被污染，它会发展出需要大量努力才能撤销的策略。每个后续决策引用被污染的内容，强化不正确的假设。

**检测和恢复**
注意症状，包括在先前成功的任务上输出质量降级、工具错位（智能体调用错误的工具或参数）、以及尽管尝试纠正仍持续的幻觉。当这些症状出现时，考虑上下文污染。

恢复需要移除或替换被污染的内容。这可能涉及将上下文截断到污染点之前、在上下文中明确注明污染并要求重新评估，或使用干净的上下文重新启动并仅保留验证的信息。

### 上下文分心

当上下文变得如此之长，以至于模型过度关注提供的信息而牺牲其训练知识时，就会出现上下文分心。模型关注上下文中提供的所有内容，无论相关性如何，这产生了即使内部知识更准确也使用提供信息的压力。

**干扰物效应**
研究表明，即使是上下文中的单个无关文档也会降低涉及相关文档的任务的性能。多个干扰物复合降级。这种效应与绝对噪声无关，而是与注意力分配有关——无关信息与相关信息竞争有限的注意力预算。

模型没有"跳过"无关上下文的机制。它们必须关注提供的所有内容，这种义务即使当无关信息明显无用时也会创建分心。

**缓解策略**
通过仔细策划进入上下文的内容来缓解分心。在加载检索的文档之前应用相关性过滤。使用命名空间和组织使无关部分在结构上易于忽略。考虑信息是否真的需要在上下文中，或者可以通过工具调用访问。

### 上下文混淆

当无关信息以降低质量的方式影响响应时，就会出现上下文混淆。这与分心有关但不同——混淆关注上下文对模型行为的影响，而不是注意力分配。

如果你在上下文中放入某些东西，模型必须关注它。模型可能纳入无关信息、使用不适当的工具定义，或应用来自不同上下文的约束。当上下文包含多个任务类型或在单个会话中在任务之间切换时，混淆尤其成问题。

**混淆的迹象**
注意响应中针对查询的错误方面、似乎适合不同任务的工具调用，或混合来自多个来源的需求的输出。这些表明对什么上下文适用于当前情况的混淆。

**架构解决方案**
架构解决方案包括不同任务获得不同上下文窗口的显式任务分段、任务上下文之间的清晰转换，以及隔离不同目标上下文的状态管理。

### 上下文冲突

当累积信息直接冲突时，就会出现上下文冲突，产生使推理脱轨的矛盾指导。这与污染不同，在污染中一条信息不正确——在冲突中，多条正确信息相互矛盾。

**冲突来源**
冲突通常来自多源检索（不同来源有矛盾信息）、版本冲突（过时和当前信息都出现在上下文中）、以及视角冲突（不同观点有效但不兼容）。

**解决方法**
解决方法包括明确冲突标记（识别矛盾并请求澄清）、优先级规则（建立哪个来源优先）、以及版本过滤（从上下文中排除过时信息）。

### 经验基准和阈值

研究提供有关降级模式的具体数据，为设计决策提供信息。

**RULER 基准发现**
RULER 基准提供了清醒的发现：声称 32K+ 上下文的模型中只有 50% 在 32K 令牌时保持满意的性能。GPT-5.2 在当前模型中显示出最少的降级，而许多模型在扩展上下文中仍下降 30+ 点。简单大海捞针测试中的接近完美分数并不能转化为真正的长上下文理解。

**模型特定降级阈值**
| 模型 | 降级开始 | 严重降级 | 说明 |
|-------|-------------------|-------------------|-------|
| GPT-5.2 | ~64K 令牌 | ~200K 令牌 | 具有思考模式的最佳整体降级抗性 |
| Claude Opus 4.5 | ~100K 令牌 | ~180K 令牌 | 200K 上下文窗口，强大的注意力管理 |
| Claude Sonnet 4.5 | ~80K 令牌 | ~150K 令牌 | 针对智能体和编码任务优化 |
| Gemini 3 Pro | ~500K 令牌 | ~800K 令牌 | 1M 上下文窗口，原生多模态 |
| Gemini 3 Flash | ~300K 令牌 | ~600K 令牌 | 比 Gemini 2.5 快 3 倍，81.2% MMMU-Pro |

**模型特定行为模式**
不同模型在上下文压力下表现出不同的故障模式：

- **Claude 4.5 系列**：具有校准不确定性的最低幻觉率。Claude Opus 4.5 在 SWE-bench Verified 上达到 80.9%。倾向于拒绝或要求澄清而不是虚构。
- **GPT-5.2**：两种模式可用 - 即时（快速）和思考（推理）。思考模式通过逐步验证减少幻觉，但增加延迟。
- **Gemini 3 Pro/Flash**：具有 1M 上下文窗口的原生多模态。Gemini 3 Flash 提供比上一代快 3 倍的速度。在文本、代码、图像、音频和视频中强大的多模态推理。

这些模式为不同用例的模型选择提供信息。高风险任务受益于 Claude 4.5 的保守方法或 GPT-5.2 的思考模式；速度关键任务可能使用即时模式。

### 反直觉发现

研究揭示了几个挑战关于上下文管理假设的反直觉模式。

**打乱的海草堆优于连贯的**
研究发现，打乱的（不连贯的）海草堆产生比逻辑连贯的更好的性能。这表明连贯的上下文可能创建混淆检索的虚假关联，而不连贯的上下文迫使模型依赖精确匹配。

**单个干扰物影响过大**
即使是单个无关文档也会显著降低性能。这种效应与噪声量不成比例，而是遵循任何干扰物存在都会触发降级的阶跃函数。

**针-问题相似性相关性**
针和问题对之间的较低相似性显示出随上下文长度更快的降级。需要在不同相似内容之间进行推理的任务特别容易受到影响。

### 更大的上下文何时有害

更大的上下文窗口并不能统一提高性能。在许多情况下，更大的上下文会产生超过收益的新问题。

**性能降级曲线**
模型表现出随上下文长度的非线性降级。性能在阈值之前保持稳定，然后快速降级。阈值因模型和任务复杂性而异。对于许多模型，有意义的降级在 8,000-16,000 令牌左右开始，即使上下文窗口支持更大的大小。

**成本影响**
处理成本随上下文长度不成比例地增长。处理 400K 令牌上下文的成本不是 200K 的两倍——它在时间和计算资源方面都呈指数增长。对于许多应用程序，这使得大上下文处理在经济上不切实际。

**认知负荷隐喻**
即使有无限的上下文，要求单个模型在数十个独立任务中保持一致质量也会创建认知瓶颈。模型必须在项目之间不断切换上下文，维护比较框架，并确保风格一致性。这不是更多上下文能解决的问题。

## 实践指导

### 四桶方法

四种策略解决上下文降级的不同方面：

**写入**：使用草稿本、文件系统或外部存储将上下文保存在窗口之外。这保持活动上下文精简，同时保留信息访问。

**选择**：通过检索、过滤和优先级将相关上下文拉入窗口。这通过排除无关信息来解决分心。

**压缩**：通过总结、抽象和观察掩码在保留信息的同时减少令牌。这扩展了有效上下文容量。

**隔离**：将上下文拆分到子智能体会话中，以防止任何单个上下文增长到足以降级的程度。这是最激进的策略，但通常是最有效的。

### 架构模式

通过特定架构模式实现这些策略。使用即时上下文加载仅在需要时检索信息。使用观察掩码用紧凑引用替换详细的工具输出。使用子智能体架构隔离不同任务的上下文。使用压缩在超过限制之前总结增长的上下文。

**示例 1：检测降级**
```yaml
# 长对话期间上下文增长
轮次_1: 1000 令牌
轮次_5: 8000 令牌
轮次_10: 25000 令牌
轮次_20: 60000 令牌（降级开始）
轮次_30: 90000 令牌（显著降级）
```

**示例 2：缓解中间丢失**
```markdown
# 用边缘的关键信息组织上下文

[当前任务]                      # 在开头
- 目标：生成季度报告
- 截止日期：周末

[详细上下文]                  # 中间（较少注意力）
- 50 页数据
- 多个分析部分
- 支持证据

[关键发现]                     # 在结尾
- 收入增长 15%
- 成本下降 8%
- A 地区增长
```

## 指南

1. 在开发期间监控上下文长度和性能相关性
2. 将关键信息放在上下文的开头或结尾
3. 在降级变得严重之前实现压缩触发器
4. 在添加到上下文之前验证检索的文档的准确性
5. 使用版本控制防止过时信息导致冲突
6. 分段任务以防止跨不同目标的上下文混淆
7. 为优雅降级设计而不是假设完美条件
8. 使用逐渐增大的上下文进行测试以找到降级阈值

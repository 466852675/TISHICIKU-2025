# 代理系统的评估方法

代理系统的评估需要与传统软件甚至标准语言模型应用不同的方法。代理做出动态决策，在不同运行之间是非确定性的，并且通常缺乏单一正确答案。有效的评估必须考虑这些特征，同时提供可操作的反馈。强大的评估框架能够实现持续改进、捕获回归，并验证上下文工程选择是否达到预期效果。

## 何时激活

在以下情况下激活此技能：
- 系统性地测试代理性能
- 验证上下文工程选择
- 随时间测量改进
- 在部署前捕获回归
- 为代理流程构建质量门
- 比较不同的代理配置
- 持续评估生产系统

## 核心概念

代理评估需要以结果为中心的方法，考虑非确定性和多个有效路径。多维度标准捕获各种质量方面：事实准确性、完整性、引用准确性、来源质量和工具效率。LLM-as-judge提供可扩展的评估，而人工评估捕获边缘案例。

关键洞察是代理可能找到实现目标的替代路径——评估应判断它们是否在遵循合理流程的同时实现了正确的结果。

**性能驱动因素：95%发现**
BrowseComp评估（测试浏览代理定位难以找到的信息的能力）的研究发现，三个因素解释了95%的性能差异：

| 因素 | 方差解释 | 含义 |
|------|----------|------|
| Token使用 | 80% | 更多token = 更好性能 |
| 工具调用次数 | ~10% | 更多探索有帮助 |
| 模型选择 | ~5% | 更好的模型倍增效率 |

这一发现对评估设计有重要影响：
- **Token预算很重要**：使用现实的token预算评估代理，而非无限资源
- **模型升级胜过token增加**：升级到Claude Sonnet 4.5或GPT-5.2比在前一版本上翻倍token预算提供更大收益
- **多代理验证**：这一发现验证了将工作分布在具有单独上下文窗口的代理中的架构

## 详细主题

### 评估挑战

**非确定性和多个有效路径**
代理可能采取完全不同的有效路径来达到目标。一个代理可能搜索三个来源，而另一个搜索十个。它们可能使用不同的工具找到相同的答案。检查特定步骤的传统评估在此上下文中失败。

解决方案是以结果为中心的评估，判断代理是否在遵循合理流程的同时实现了正确的结果。

**上下文相关的失败**
代理失败通常以微妙的方式依赖于上下文。代理可能在简单查询上成功，但在复杂查询上失败。它可能在一组工具上工作良好，但在另一组上失败。失败可能仅在扩展交互后上下文累积时才出现。

评估必须涵盖一系列复杂性级别，并测试扩展交互，而不仅仅是孤立查询。

**复合质量维度**
代理质量不是单一维度。它包括事实准确性、完整性、连贯性、工具效率和流程质量。代理可能在准确性上得分高但在效率上得分低，反之亦然。

评估标准必须捕获具有适当权重的多个维度，以适应用例。

### 评估标准设计

**多维度标准**
有效的标准涵盖具有描述性级别的关键维度：

事实准确性：声明与事实匹配（优秀到失败）

完整性：输出涵盖请求的方面（优秀到失败）

引用准确性：引用与声称的来源匹配（优秀到失败）

来源质量：使用适当的主要来源（优秀到失败）

工具效率：合理次数使用正确的工具（优秀到失败）

**标准评分**
将维度评估转换为数字分数（0.0到1.0）并适当加权。计算加权总体分数。根据用例要求确定通过阈值。

### 评估方法论

**LLM-as-Judge**
基于LLM的评估可扩展到大型测试集并提供一致的判断。关键是设计捕获感兴趣维度的有效评估提示。

提供清晰的任务描述、代理输出、事实（如果可用）、具有级别描述的评估量表，并请求结构化判断。

**人工评估**
人工评估捕获自动化遗漏的内容。人类注意到不寻常查询上的幻觉答案、系统故障和自动化评估遗漏的微妙偏见。

有效的人工评估涵盖边缘案例、系统采样、跟踪模式，并提供情境理解。

**最终状态评估**
对于改变持久状态的代理，最终状态评估关注最终状态是否符合期望，而非代理如何达到那里。

### 测试集设计

**样本选择**
在开发期间从小样本开始。在代理开发早期，变化具有戏剧性影响，因为有丰富的低 hanging fruit。小测试集揭示大效应。

从实际使用模式中采样。添加已知的边缘案例。确保跨复杂性级别的覆盖。

**复杂性分层**
测试集应跨越复杂性级别：简单（单次工具调用）、中等（多次工具调用）、复杂（多次工具调用、显著模糊性）和非常复杂（扩展交互、深度推理）。

### 上下文工程评估

**测试上下文策略**
应通过系统评估验证上下文工程选择。在不同上下文策略上运行相同测试集的代理。比较质量分数、token使用和效率指标。

**退化测试**
通过在不同上下文大小上运行代理来测试上下文退化如何影响性能。识别上下文成为问题的性能悬崖。建立安全操作限制。

### 持续评估

**评估流程**
构建在代理更改时自动运行的评估流程。随时间跟踪结果。比较版本以识别改进或回归。

**监控生产**
通过采样交互并随机评估来跟踪生产中的评估指标。为质量下降设置警报。维护趋势分析仪表板。

## 实用指导

### 构建评估框架

1. 定义与您的用例相关的质量维度
2. 创建具有清晰、可操作级别描述的标准
3. 从实际使用模式和边缘案例构建测试集
4. 实施自动化评估流程
5. 在进行更改之前建立基线指标
6. 在所有重大更改上运行评估
7. 随时间跟踪指标以进行趋势分析
8. 用人工审查补充自动化评估

### 避免评估陷阱

过度拟合特定路径：评估结果，而非特定步骤。
忽略边缘案例：包括多样化的测试场景。
单一指标痴迷：使用多维度标准。
忽视上下文效应：使用现实的上下文大小进行测试。
跳过人工评估：自动化评估遗漏微妙问题。

def evaluate_agent_response(response, expected):
    rubric = load_rubric()
    scores = {}
    for dimension, config in rubric.items():
        scores[dimension] = assess_dimension(response, expected, dimension)
    overall = weighted_average(scores, config["weights"])
    return {"passed": overall >= 0.7, "scores": scores}

## 指南

1. 使用多维度标准，而非单一指标
2. 评估结果，而非特定执行路径
3. 从简单到复杂涵盖复杂性级别
4. 使用现实的上下文大小和历史进行测试
5. 持续运行评估，而非仅在发布前
6. 用人工审查补充LLM评估
7. 随时间跟踪指标以进行检测
8. 根据用例设置清晰的通过/失败阈值

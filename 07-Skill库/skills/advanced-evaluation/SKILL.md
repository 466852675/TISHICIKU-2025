# 高级评估

本技能涵盖使用LLM作为评判者来评估LLM输出的生产级技术。它综合了学术研究、行业实践和实际实施经验，为构建可靠的评估系统提供可行的模式。

**核心洞察**：LLM作为评判者并非单一技术，而是一系列方法，每种方法适用于不同的评估场景。选择正确的方法并缓解已知的偏见是本技能培养的核心能力。

## 何时激活

在以下情况下激活此技能：

- 为LLM输出构建自动化评估流程
- 比较多个模型响应以选择最佳方案
- 在评估团队中建立一致的质量标准
- 调试显示不一致结果的评估系统
- 设计提示或模型变更的A/B测试
- 创建人工或自动化评估的标准
- 分析自动化评估与人工判断之间的相关性

## 核心概念

### 评估分类法

评估方法分为两大类，具有不同的可靠性特征：

**直接评分**：单个LLM按定义的量表对单个响应进行评分。
- 最适合：客观标准（事实准确性、指令遵循、毒性）
- 可靠性：对于定义明确的标准为中等至高等
- 失败模式：评分校准漂移、量表解释不一致

**成对比较**：LLM比较两个响应并选择更好的一个。
- 最适合：主观偏好（语气、风格、说服力）
- 可靠性：对于偏好评估高于直接评分
- 失败模式：位置偏见、长度偏见

MT-Bench论文（Zheng等人，2023）的研究表明，对于基于偏好的评估，成对比较比直接评分更能与人类评判者达成一致，而对于具有明确真实答案的客观标准，直接评分仍然适用。

### 偏见图景

LLM评判者表现出必须主动缓解的系统性偏见：

**位置偏见**：在成对比较中，第一位置的响应获得优先待遇。缓解方法：交换位置评估两次，使用多数投票或一致性检查。

**长度偏见**：无论质量如何，较长的响应获得更高评分。缓解方法：明确提示忽略长度，使用长度归一化评分。

**自我增强偏见**：模型对自己输出的评分更高。缓解方法：使用不同模型进行生成和评估，或承认此限制。

**冗长偏见**：详细的解释获得更高分数，即使不必要。缓解方法：针对具体标准的评分标准，惩罚不相关的细节。

**权威偏见**：自信、权威的语气无论准确性如何都会获得更高评分。缓解方法：要求引用证据，增加事实核查层。

### 指标选择框架

根据评估任务结构选择指标：

| 任务类型 | 主要指标 | 次要指标 |
|-----------|-----------------|-------------------|
| 二元分类（通过/失败） | 召回率、精确率、F1 | Cohen's κ |
| 序数量表（1-5评分） | Spearman's ρ、Kendall's τ | Cohen's κ（加权） |
| 成对偏好 | 一致率、位置一致性 | 置信度校准 |
| 多标签 | Macro-F1、Micro-F1 | 每标签精确率/召回率 |

关键洞察：高绝对一致性不如系统性分歧模式重要。在特定标准上与人类持续不一致的评判者比具有随机噪声的评判者问题更大。

## 评估方法

### 直接评分实施

直接评分需要三个组件：明确的标准、校准的量表和结构化的输出格式。

**标准定义模式**：
```
标准：[名称]
描述：[此标准衡量的内容]
权重：[相对重要性，0-1]
```

**量表校准**：
- 1-3量表：二元加中性选项，认知负荷最低
- 1-5量表：标准Likert量表，粒度与可靠性的良好平衡
- 1-10量表：高粒度但难以校准，仅在详细评分标准时使用

**直接评分的提示结构**：
```
您是评估响应质量的专家评估员。

## 任务
根据每个标准评估以下响应。

## 原始提示
{prompt}

## 待评估的响应
{response}

## 标准
{每个标准：名称、描述、权重}

## 说明
对于每个标准：
1. 在响应中找到具体证据
2. 根据评分标准评分（1-{max}量表）
3. 用证据证明您的评分
4. 提出一个具体的改进建议

## 输出格式
以包含分数、理由和摘要的结构化JSON响应。
```

**思维链要求**：所有评分提示必须要求在评分之前提供理由。研究表明，这比先评分的方法提高了15-25%的可靠性。

### 成对比较实施

成对比较对于基于偏好的评估本质上更可靠，但需要偏见缓解。

**位置偏见缓解协议**：
1. 第一次：响应A在第一位置，响应B在第二位置
2. 第二次：响应B在第一位置，响应A在第二位置
3. 一致性检查：如果两次结果不一致，返回平局并降低置信度
4. 最终裁决：一致的获胜者，取平均置信度

**成对比较的提示结构**：
```
您是比较两个AI响应的专家评估员。

## 关键说明
- 不要因为响应更长而偏好它们
- 不要基于位置（第一vs第二）偏好响应
- 仅根据指定标准关注质量
- 当响应真正等同时，平局是可以接受的

## 原始提示
{prompt}

## 响应A
{response_a}

## 响应B
{response_b}

## 比较标准
{标准列表}

## 说明
1. 首先独立分析每个响应
2. 在每个标准上比较它们
3. 确定总体获胜者及置信度水平

## 输出格式
包含每标准比较、总体获胜者、置信度（0-1）和推理的JSON。
```

**置信度校准**：置信度分数应反映位置一致性：
- 两次通过一致：置信度 = 个体置信度的平均值
- 通过不一致：置信度 = 0.5，裁决 = 平局

### 评分标准生成

定义明确的评分标准比开放式评分减少40-60%的评估方差。

**评分标准组件**：
1. **级别描述**：每个分数级别的清晰边界
2. **特征**：定义每个级别的可观察特征
3. **示例**：每个级别的代表性文本（可选但有价值）
4. **边缘情况**：对模糊情况的指导
5. **评分指南**：一致应用的一般原则

**严格度校准**：
- **宽松**：通过分数的门槛较低，适合鼓励迭代
- **平衡**：公平、典型的生产使用期望
- **严格**：高标准，适合安全关键或高风险的评估

**领域适配**：评分标准应使用领域特定术语。"代码可读性"评分标准提及变量、函数和注释。"医疗准确性"评分标准引用临床术语和证据标准。

## 实用指导

### 评估流程设计

生产评估系统需要多层架构：

```
┌─────────────────────────────────────────────────┐
│                 评估流程                          │
├─────────────────────────────────────────────────┤
│                                                   │
│  输入：响应 + 提示 + 上下文                        │
│           │                                       │
│           ▼                                       │
│  ┌─────────────────────┐                         │
│  │   标准加载器         │ ◄── 评分标准、权重      │
│  └──────────┬──────────┘                         │
│             │                                     │
│             ▼                                     │
│  ┌─────────────────────┐                         │
│  │   主要评分器         │ ◄── 直接或成对          │
│  └──────────┬──────────┘                         │
│             │                                     │
│             ▼                                     │
│  ┌─────────────────────┐                         │
│  │   偏见缓解           │ ◄── 位置交换等          │
│  └──────────┬──────────┘                         │
│             │                                     │
│             ▼                                     │
│  ┌─────────────────────┐                         │
│  │ 置信度评分           │ ◄── 校准                │
│  └──────────┬──────────┘                         │
│             │                                     │
│             ▼                                     │
│  输出：分数 + 理由 + 置信度                        │
│                                                   │
└─────────────────────────────────────────────────┘
```

### 常见反模式

**反模式：无理由评分**
- 问题：分数缺乏依据，难以调试或改进
- 解决方案：始终要求在评分之前提供基于证据的理由

**反模式：单次通过成对比较**
- 问题：位置偏见破坏结果
- 解决方案：始终交换位置并检查一致性

**反模式：过载的标准**
- 问题：衡量多个事物的标准不可靠
- 解决方案：一个标准 = 一个可衡量的方面

**反模式：缺少边缘情况指导**
- 问题：评估员不一致地处理模糊情况
- 解决方案：在评分标准中包含边缘情况及明确指导

**反模式：忽略置信度校准**
- 问题：高置信度的错误判断比低置信度更糟糕
- 解决方案：将置信度校准到位置一致性和证据强度

### 决策框架：直接 vs 成对

使用此决策树：

```
是否存在客观真实答案？
├── 是 → 直接评分
│   └── 示例：事实准确性、指令遵循、格式合规
│
└── 否 → 是偏好或质量判断？
    ├── 是 → 成对比较
    │   └── 示例：语气、风格、说服力、创造力
    │
    └── 否 → 考虑基于参考的评估
        └── 示例：摘要（与来源比较）、翻译（与参考比较）
```

### 扩展评估

对于高容量评估：

1. **LLM小组（PoLL）**：使用多个模型作为评判者，聚合投票
   - 减少个体模型偏见
   - 更昂贵但对高风险决策更可靠

2. **分层评估**：快速廉价模型进行筛选，昂贵模型处理边缘情况
   - 对大量数据成本效益高
   - 需要校准筛选阈值

3. **人机协作**：明确案例使用自动化评估，低置信度案例使用人工审核
   - 对关键应用可靠性最高
   - 设计反馈循环以改进自动化评估

## 示例

### 示例1：准确性的直接评分

**输入**：
```
提示："什么导致地球上的季节变化？"
响应："季节是由地球倾斜的轴引起的。当地球绕太阳运行时，
不同半球在不同时间接收更直接的阳光。"
标准：事实准确性（权重：1.0）
量表：1-5
```

**输出**：
```json
{
  "criterion": "事实准确性",
  "score": 5,
  "evidence": [
    "正确识别地轴倾斜是主要原因",
    "正确解释半球间阳光分布差异",
    "没有事实错误"
  ],
  "justification": "响应准确地用正确的科学原理解释了季节的原因。地轴倾斜及其对阳光分布的影响都被正确描述。",
  "improvement": "可以添加具体的倾斜角度（23.5°）以使其更完整。"
}
```

### 示例2：带位置交换的成对比较

**输入**：
```
提示："向初学者解释机器学习"
响应A：[带有术语的技术解释]
响应B：[基于简单类比的解释]
标准：["清晰度"、"可访问性"]
```

**第一次通过（A在前）**：
```json
{ "winner": "B", "confidence": 0.8 }
```

**第二次通过（B在前）**：
```json
{ "winner": "A", "confidence": 0.6 }
```
（注意：获胜者是A，因为B在第一位置）

**映射后的第二次通过**：
```json
{ "winner": "B", "confidence": 0.6 }
```

**最终结果**：
```json
{
  "winner": "B",
  "confidence": 0.7,
  "positionConsistency": {
    "consistent": true,
    "firstPassWinner": "B",
    "secondPassWinner": "B"
  }
}
```

## 指南

1. **始终要求在评分之前提供理由** - 思维链提示将可靠性提高15-25%

2. **始终在成对比较中交换位置** - 单次通过比较会被位置偏见破坏

3. **将量表粒度与评分标准特异性匹配** - 没有详细级别描述时不要使用1-10量表

4. **分离客观和主观标准** - 对客观标准使用直接评分，对主观标准使用成对比较

5. **包含置信度分数** - 校准到位置一致性和证据强度

6. **明确定义边缘情况** - 模糊情况导致最多的评估方差

7. **使用领域特定的评分标准** - 通用评分标准产生通用（不太有用）的评估

8. **针对人工判断进行验证** - 自动化评估只有与人工评估相关时才有价值

9. **监控系统性偏见** - 按标准、响应类型、模型跟踪分歧模式

10. **为迭代而设计** - 评估系统通过反馈循环改进

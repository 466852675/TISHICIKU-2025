# 基于文件系统的上下文工程

文件系统提供了一个单一接口，智能体可以通过它灵活地存储、检索和更新实际上无限量的上下文。这种模式解决了上下文窗口有限而任务通常需要比单个窗口能容纳的更多信息的根本约束。

核心洞察是文件支持动态上下文发现：智能体按需拉取相关上下文，而不是在上下文窗口中携带所有内容。这与静态上下文形成对比，后者无论相关性如何始终包含在内。

## 何时激活

在以下情况下激活此技能：
- 工具输出正在膨胀上下文窗口
- 智能体需要在长轨迹中持久化状态
- 子智能体必须共享信息而无需直接消息传递
- 任务需要比窗口能容纳的更多上下文
- 构建学习和更新自己指令的智能体
- 实现中间结果的草稿本
- 终端输出或日志需要可被智能体访问

## 核心概念

上下文工程可能以四种可预测的方式失败。首先，当智能体需要的上下文不在总可用上下文中时。其次，当检索到的上下文未能封装所需上下文时。第三，当检索到的上下文远超所需上下文时，浪费令牌并降低性能。第四，当智能体无法发现埋藏在许多文件中的小众信息时。

文件系统通过提供持久层来解决这些失败，智能体在其中写入一次并选择性读取，卸载批量内容同时保留通过搜索工具检索特定信息的能力。

## 详细主题

### 静态与动态上下文权衡

**静态上下文**
静态上下文始终包含在提示中：系统指令、工具定义和关键规则。静态上下文无论任务相关性如何都会消耗令牌。随着智能体积累更多能力（工具、技能、指令），静态上下文增长并挤占动态信息的空间。

**动态上下文发现**
动态上下文在相关时按需加载。智能体接收最小的静态指针（名称、描述、文件路径），并在需要时使用搜索工具加载完整内容。

动态发现在令牌效率方面更高，因为只有必要的数据进入上下文窗口。它还可以通过减少潜在混淆或矛盾的信息来提高响应质量。

权衡：动态发现需要模型正确识别何时加载额外上下文。这在当前前沿模型中运行良好，但可能在无法识别何时需要更多信息的较弱模型中失败。

### 模式 1：文件系统作为草稿本

**问题**
工具调用可能返回大量输出。网络搜索可能返回 10k 令牌的原始内容。数据库查询可能返回数百行。如果此内容进入消息历史，它将在整个对话中保留，膨胀令牌成本并可能降低对更相关信息的注意力。

**解决方案**
将大型工具输出写入文件，而不是直接返回到上下文。然后智能体使用有针对性的检索（grep、特定行读取）来提取仅相关部分。

**实现**
```python
def handle_tool_output(output: str, threshold: int = 2000) -> str:
    if len(output) < threshold:
        return output
    
    # 写入草稿本
    file_path = f"scratch/{tool_name}_{timestamp}.txt"
    write_file(file_path, output)
    
    # 返回引用而不是内容
    key_summary = extract_summary(output, max_tokens=200)
    return f"[输出写入 {file_path}。摘要：{key_summary}]"
```

智能体然后可以使用 `grep` 搜索特定模式或 `read_file` 配合行范围来检索目标部分。

**好处**
- 减少长对话中的令牌累积
- 保留完整输出供以后参考
- 支持有针对性的检索而不是携带所有内容

### 模式 2：计划持久化

**问题**
长周期任务要求智能体制定计划并遵循它们。但随着对话延长，计划可能失去注意力或被总结丢失。智能体失去对应该做什么的跟踪。

**解决方案**
将计划写入文件系统。智能体可以在任何时候重新阅读其计划，提醒自己当前目标和进展。这有时被称为"通过背诵操纵注意力"。

**实现**
以结构化格式存储计划：
```yaml
# scratch/current_plan.yaml
objective: "重构认证模块"
status: 进行中
steps:
  - id: 1
    description: "审计当前认证端点"
    status: 已完成
  - id: 2
    description: "设计新的令牌验证流程"
    status: 进行中
  - id: 3
    description: "实现和测试更改"
    status: 待处理
```

智能体在每次轮次开始时或在需要重新定向时读取此文件。

### 模式 3：通过文件系统的子智能体通信

**问题**
在多智能体系统中，子智能体通常通过消息传递向协调器智能体报告发现。这创建了一个"传话游戏"，信息在每一跳通过总结而降级。

**解决方案**
子智能体直接将发现写入文件系统。协调器直接读取这些文件，绕过中间消息传递。这保留保真度并减少协调器中的上下文累积。

**实现**
```
workspace/
  agents/
    research_agent/
      findings.md        # 研究智能体写入此处
      sources.jsonl      # 来源跟踪
    code_agent/
      changes.md         # 代码智能体写入此处
      test_results.txt   # 测试输出
  coordinator/
    synthesis.md         # 协调器读取智能体输出，写入综合
```

每个智能体在相对隔离中操作，但通过文件系统共享状态。

### 模式 4：动态技能加载

**问题**
智能体可能有许多技能或指令集，但大多数对任何给定任务都是无关的。将所有指令塞入系统提示会浪费令牌并可能使模型与矛盾或无关的指导混淆。

**解决方案**
将技能存储为文件。仅在静态上下文中包含技能名称和简要描述。智能体在任务需要时使用搜索工具加载相关技能内容。

**实现**
静态上下文包括：
```
可用技能（相关时使用 read_file 加载）：
- database-optimization: 查询调优和索引策略
- api-design: REST/GraphQL 最佳实践
- testing-strategies: 单元、集成和 e2e 测试模式
```

智能体仅在处理数据库任务时加载 `skills/database-optimization/SKILL.md`。

### 模式 5：终端和日志持久化

**问题**
来自长时间运行进程的终端输出快速累积。将输出复制并粘贴到智能体输入中是手动的且效率低下。

**解决方案**
自动将终端输出同步到文件。然后智能体可以 grep 相关部分（错误消息、特定命令），而无需加载整个终端历史。

**实现**
终端会话持久化为文件：
```
terminals/
  1.txt    # 终端会话 1 输出
  2.txt    # 终端会话 2 输出
```

智能体使用有针对性的 grep 查询：
```bash
grep -A 5 "error" terminals/1.txt
```

### 模式 6：通过自我修改学习

**问题**
智能体通常缺乏用户在交互中隐式或显式提供的上下文。传统上，这需要会话之间手动更新系统提示。

**解决方案**
智能体将学习到的信息写入自己的指令文件。后续会话加载这些文件，自动整合学习到的上下文。

**实现**
用户提供偏好后：
```python
def remember_preference(key: str, value: str):
    preferences_file = "agent/user_preferences.yaml"
    prefs = load_yaml(preferences_file)
    prefs[key] = value
    write_yaml(preferences_file, prefs)
```

后续会话包括一步加载用户偏好（如果文件存在）。

**注意**
这种模式仍在出现。自我修改需要仔细的防护措施，以防止智能体随时间积累不正确或矛盾的指令。

### 文件系统搜索技术

模型专门训练以理解文件系统遍历。`ls`、`glob`、`grep` 和带行范围的 `read_file` 组合提供强大的上下文发现：

- `ls` / `list_dir`: 发现目录结构
- `glob`: 查找匹配模式的文件（例如 `**/*.py`）
- `grep`: 搜索文件内容中的模式，返回匹配行
- `read_file` 配合范围：读取特定行范围而不加载整个文件

这种组合通常在技术内容（代码、API 文档）上优于语义搜索，其中语义意义稀疏但结构模式清晰。

语义搜索和文件系统搜索很好地协同工作：语义搜索用于概念查询，文件系统搜索用于结构和精确匹配查询。

## 实践指导

### 何时使用文件系统上下文

**在以下情况下使用文件系统模式：**
- 工具输出超过 2000 令牌
- 任务跨越多个对话轮次
- 多个智能体需要共享状态
- 技能或指令超过系统提示中舒适容纳的范围
- 日志或终端输出需要有选择地查询

**在以下情况下避免文件系统模式：**
- 任务在单轮次中完成
- 上下文舒适地适合窗口
- 延迟至关重要（文件 I/O 增加开销）
- 简单模型无法使用文件系统工具

### 文件组织

为可发现性构建文件：
```
project/
  scratch/           # 临时工作文件
    tool_outputs/    # 大型工具结果
    plans/           # 活动计划和清单
  memory/            # 持久化学习信息
    preferences.yaml # 用户偏好
    patterns.md      # 学习到的模式
  skills/            # 可加载的技能定义
  agents/            # 子智能体工作区
```

使用一致的命名约定。在草稿文件中包含时间戳或 ID 以消除歧义。

### 令牌核算

跟踪令牌来源：
- 测量静态与动态上下文比率
- 监控卸载前后的工具输出大小
- 跟踪动态上下文实际加载的频率

基于测量优化，而不是假设。

## 示例

**示例 1：工具输出卸载**
```
输入：网络搜索返回 8000 令牌
之前：8000 令牌添加到消息历史
之后：
  - 写入 scratch/search_results_001.txt
  - 返回："[结果在 scratch/search_results_001.txt。关键发现：API 速率限制为 1000 请求/分钟]"
  - 智能体在需要特定细节时 grep 文件
结果：上下文中约 100 令牌，8000 令牌按需访问
```

**示例 2：动态技能加载**
```
输入：用户询问数据库索引
静态上下文："database-optimization: 查询调优和索引"
智能体操作：read_file("skills/database-optimization/SKILL.md")
结果：仅在相关时加载完整技能
```

**示例 3：聊天历史作为文件引用**
```
触发：达到上下文窗口限制，需要总结
操作：
  1. 将完整历史写入 history/session_001.txt
  2. 为新上下文窗口生成摘要
  3. 包含引用："完整历史在 history/session_001.txt"
结果：智能体可以搜索历史文件以恢复总结中丢失的细节
```

## 指南

1. 将大型输出写入文件；将摘要和引用返回到上下文
2. 在结构化文件中存储计划和状态以供重新阅读
3. 使用子智能体文件工作区代替消息链
4. 动态加载技能而不是将所有技能塞入系统提示
5. 将终端和日志输出持久化为可搜索文件
6. 将 grep/glob 与语义搜索结合以进行全面发现
7. 为智能体可发现性组织文件，使用清晰的命名
8. 测量令牌节省以验证文件系统模式有效
9. 为草稿文件实现清理以防止无界增长
10. 用验证保护自我修改模式

# 参考：Manus 上下文工程原则

此技能基于 Manus 的上下文工程原则，Manus 是于 2025 年 12 月被 Meta 以 20 亿美元收购的 AI 代理公司。

## 6 条 Manus 原则

### 原则 1：围绕 KV-Cache 设计

> "KV-cache 命中率是生产级 AI 代理的**唯一**最重要指标。"

**统计数据：**
- 约 100:1 的输入输出 token 比例
- 缓存 token：$0.30/MTok vs 未缓存：$3/MTok
- 10 倍成本差异！

**实现：**
- 保持提示前缀**稳定**（单 token 更改会使缓存失效）
- 系统提示中**没有时间戳**
- 使上下文**仅追加**并具有确定性序列化

### 原则 2：掩码，不要移除

不要动态移除工具（会破坏 KV-cache）。改用 logit 掩码。

**最佳实践：** 使用一致的动作前缀（例如，`browser_`、`shell_`、`file_`）以便于掩码。

### 原则 3：文件系统作为外部记忆

> "Markdown 是我在磁盘上的'工作记忆'。"

**公式：**
```
上下文窗口 = RAM（易失性，有限）
文件系统 = 磁盘（持久性，无限）
```

**压缩必须是可恢复的：**
- 即使丢弃网页内容也要保留 URL
- 丢弃文档内容时保留文件路径
- 永远不要丢失指向完整数据的指针

### 原则 4：通过背诵操纵注意力

> "在整个任务过程中创建和更新 todo.md，将全局计划推入模型的近期注意力范围。"

**问题：** 约 50 个工具调用后，模型会忘记原始目标（"迷失在中间"效应）。

**解决方案：** 在每次决策前重新阅读 `task_plan.md`。目标出现在注意力窗口中。

```
上下文开始：[原始目标 - 很远，被遗忘]
...许多工具调用...
上下文结束：[最近阅读的 task_plan.md - 获得注意力！]
```

### 原则 5：保留错误的内容

> "将错误的内容保留在上下文中。"

**原因：**
- 带有堆栈跟踪的失败操作让模型隐式更新信念
- 减少错误重复
- 错误恢复是"真正代理行为的**最清晰信号**之一"

### 原则 6：不要被 Few-Shot

> "统一孕育脆弱。"

**问题：** 重复的动作-观察对导致漂移和幻觉。

**解决方案：** 引入受控变化：
- 稍微改变措辞
- 不要盲目复制粘贴模式
- 在重复任务上重新校准

## 3 种上下文工程策略

基于 Lance Martin 对 Manus 架构的分析。

### 策略 1：上下文缩减

**压缩：**
```
工具调用有两种表示：
├── 完整：原始工具内容（存储在文件系统中）
└── 紧凑：仅引用/文件路径

规则：
- 对陈旧（较旧）的工具结果应用压缩
- 保持最近结果完整（以指导下一个决策）
```

**摘要：**
- 当压缩达到收益递减时应用
- 使用完整工具结果生成
- 创建标准化摘要对象

### 策略 2：上下文隔离（多代理）

**架构：**
```
┌─────────────────────────────────┐
│         规划器代理              │
│  └─ 分配任务给子代理            │
├─────────────────────────────────┤
│       知识管理器                │
│  └─ 审查对话                    │
│  └─ 确定文件系统存储            │
├─────────────────────────────────┤
│      执行器子代理               │
│  └─ 执行分配的任务              │
│  └─ 拥有自己的上下文窗口        │
└─────────────────────────────────┘
```

**关键洞察：** Manus 最初使用 `todo.md` 进行任务规划，但发现约 33% 的操作用于更新它。转向专用规划器代理调用执行器子代理。

### 策略 3：上下文卸载

**工具设计：**
- 总共使用 <20 个原子函数
- 将完整结果存储在文件系统中，而不是上下文中
- 使用 `glob` 和 `grep` 进行搜索
- 渐进式披露：仅在需要时加载信息

## 代理循环

Manus 在连续 7 步循环中运行：

```
┌─────────────────────────────────────────┐
│  1. 分析上下文                          │
│     - 理解用户意图                      │
│     - 评估当前状态                      │
│     - 审查最近的观察                    │
├─────────────────────────────────────────┤
│  2. 思考                                │
│     - 我应该更新计划吗？                │
│     - 下一个逻辑动作是什么？            │
│     - 有阻塞吗？                        │
├─────────────────────────────────────────┤
│  3. 选择工具                            │
│     - 选择一个工具                      │
│     - 确保参数可用                      │
├─────────────────────────────────────────┤
│  4. 执行动作                            │
│     - 工具在沙箱中运行                  │
├─────────────────────────────────────────┤
│  5. 接收观察                            │
│     - 结果追加到上下文                  │
├─────────────────────────────────────────┤
│  6. 迭代                                │
│     - 返回步骤 1                        │
│     - 继续直到完成                      │
├─────────────────────────────────────────┤
│  7. 交付结果                            │
│     - 将结果发送给用户                  │
│     - 附加所有相关文件                  │
└─────────────────────────────────────────┘
```

## Manus 创建的文件类型

| 文件 | 目的 | 何时创建 | 何时更新 |
|------|---------|--------------|--------------|
| `task_plan.md` | 阶段跟踪、进度 | 任务开始 | 完成阶段后 |
| `findings.md` | 发现、决策 | 任何发现后 | 查看图像/PDF 后 |
| `progress.md` | 会话日志、已完成内容 | 在断点处 | 整个会话期间 |
| 代码文件 | 实现 | 执行前 | 错误后 |

---

## 关键约束

- **单动作执行：** 每轮一个工具调用。无并行执行。
- **计划是必需的：** 代理必须始终知道：目标、当前阶段、剩余阶段
- **文件是记忆：** 上下文 = 易失性。文件系统 = 持久性。
- **永远不要重复失败：** 如果动作失败，下一个动作必须不同
- **通信是一种工具：** 消息类型：`info`（进度）、`ask`（阻塞）、`result`（终端）

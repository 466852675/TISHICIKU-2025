# 上下文工程基础

上下文是推理时可用的完整状态。它包括模型在生成响应时可以关注的所有内容：系统指令、工具定义、检索的文档、消息历史和工具输出。理解上下文基础是有效上下文工程的先决条件。

## 何时激活

在以下情况下激活此技能：
- 设计新的智能体系统或修改现有架构
- 调试可能与上下文相关的意外智能体行为
- 优化上下文使用以降低成本或提高性能
- 让新团队成员了解上下文工程概念
- 审查与上下文相关的设计决策

## 核心概念

上下文包含几个不同的组件，每个组件具有不同的特征和约束。注意力机制创建了约束有效上下文使用的有限预算。渐进式披露通过仅在需要时加载信息来管理此约束。工程学科是策划实现期望结果的最小高信号令牌集。

## 详细主题

### 上下文的解剖

**系统提示**
系统提示建立智能体的核心身份、约束和行为准则。它们在会话开始时加载一次，通常在整个对话中持续存在。系统提示应该非常清晰，使用简单、直接的语言，处于适合智能体的正确高度。

正确的高度平衡两种失败模式。在一个极端，工程师硬编码复杂的脆弱逻辑，造成脆弱性和维护负担。在另一个极端，工程师提供模糊的高层指导，无法为期望的输出给出具体信号，或错误地假设共享上下文。最佳高度取得平衡：足够具体以有效指导行为，但足够灵活以提供强大的启发式。

使用 XML 标记或 Markdown 标题将提示组织成不同的部分，以区分背景信息、指令、工具指导和输出描述。确切的格式随着模型能力的提升而变得不那么重要，但结构清晰仍然有价值。

**工具定义**
工具定义指定智能体可以采取的操作。每个工具包括名称、描述、参数和返回格式。工具定义在序列化后位于上下文前部附近，通常在系统提示之前或之后。

工具描述共同引导智能体行为。糟糕的描述迫使智能体猜测；优化的描述包括使用上下文、示例和默认值。整合原则指出，如果人类工程师无法明确说出在给定情况下应该使用哪个工具，就不能期望智能体做得更好。

**检索的文档**
检索的文档提供特定领域的知识、参考材料或任务相关信息。智能体使用检索增强生成在运行时将相关文档拉入上下文，而不是预加载所有可能的信息。

即时方法维护轻量级标识符（文件路径、存储的查询、网络链接），并使用这些引用动态加载数据。这反映了人类认知：我们通常不记忆整个信息库，而是使用外部组织和索引系统按需检索相关信息。

**消息历史**
消息历史包含用户和智能体之间的对话，包括以前的查询、响应和推理。对于长时间运行的任务，消息历史可能增长到主导上下文使用。

消息历史充当草稿本内存，智能体在其中跟踪进度、维护任务状态并在轮次之间保留推理。有效管理消息历史对于长周期任务完成至关重要。

**工具输出**
工具输出是智能体操作的结果：文件内容、搜索结果、命令执行输出、API 响应和类似数据。工具输出构成典型智能体轨迹中的大部分令牌，研究表明观察（工具输出）可能达到总上下文使用的 83.9%。

工具输出无论与当前决策相关与否都会消耗上下文。这产生了对观察掩码、压缩和选择性工具结果保留等策略的压力。

### 上下文窗口和注意力机制

**注意力预算约束**
语言模型通过注意力机制处理令牌，该机制在上下文中的所有令牌之间创建成对关系。对于 n 个令牌，这会创建必须计算和存储的 n² 个关系。随着上下文长度增加，模型捕获这些关系的能力被拉伸。

模型从训练数据分布中发展注意力模式，其中较短序列占主导地位。这意味着模型对上下文范围的依赖关系和专门参数的经验较少。结果是随着上下文增长而耗尽的"注意力预算"。

**位置编码和上下文扩展**
位置编码插值允许模型通过将它们适应最初训练的较小上下文来处理更长的序列。然而，这种适应引入了令牌位置理解的降级。模型在更长的上下文中仍然保持高能力，但与较短上下文上的性能相比，在信息检索和长距离推理方面显示出降低的精度。

**渐进式披露原则**
渐进式披露通过仅在需要时加载信息来有效管理上下文。在启动时，智能体仅加载技能名称和描述——足以知道技能何时可能相关。完整内容仅在针对特定任务激活技能时加载。

这种方法保持智能体快速，同时使它们能够按需访问更多上下文。该原则适用于多个层面：技能选择、文档加载，甚至工具结果检索。

### 上下文质量与上下文数量

假设更大的上下文窗口解决内存问题已被经验证伪。上下文工程意味着找到最小可能的高信号令牌集，以最大化期望结果的可能性。

几个因素产生了上下文效率的压力。处理成本随上下文长度不成比例地增长——不仅仅是双倍的令牌双倍成本，而是在时间和计算资源方面呈指数增长。即使使用前缀缓存，模型性能在某些上下文长度之后也会降级。长输入仍然昂贵。

指导原则是信息性优于详尽性。包括手头决策相关的内容，排除不相关的内容，并设计可以按需访问额外信息的系统。

### 上下文作为有限资源

上下文必须被视为具有递减边际回报的有限资源。就像具有有限工作内存的人类一样，语言模型在解析大量上下文时会消耗注意力预算。

引入的每个新令牌都会消耗一定量的预算。这需要仔细管理可用令牌。工程问题是在固有约束下优化效用。

上下文工程是迭代的，策划阶段发生在每次你决定传递给模型什么时。它不是一次性的提示编写练习，而是持续的上下文管理纪律。

## 实践指导

### 基于文件系统的访问

具有文件系统访问权限的智能体可以自然地使用渐进式披露。存储参考资料、文档和数据。仅在需要时使用标准文件系统操作加载文件。这种模式避免了用可能不相关的信息填充上下文。

文件系统本身提供智能体可以导航的结构。文件大小暗示复杂性；命名约定暗示目的；时间戳作为相关性的代理。文件引用的元数据提供了一种有效优化行为的机制。

### 混合策略

最有效的智能体采用混合策略。预加载一些上下文以提高速度（如 CLAUDE.md 文件或项目规则），但使智能体能够根据需要自主探索额外的上下文。决策边界取决于任务特征和上下文动态。

对于动态内容较少的上下文，预加载更多是有意义的。对于快速变化或高度特定的信息，即时加载避免陈旧上下文。

### 上下文预算

设计时考虑显式上下文预算。了解模型和任务的有效上下文限制。在开发期间监控上下文使用。在适当阈值实现压缩触发器。假设上下文会降级而不是希望它不会，来设计系统。

有效的上下文预算需要理解的不仅仅是原始令牌计数，还有注意力分布模式。上下文中间接收的注意力比开头和结尾少。将关键信息放在受青睐的注意力位置。

## 示例

**示例 1：组织系统提示**
```markdown
<背景信息>
你是帮助开发团队的 Python 专家。
当前项目：Python 3.9+ 中的数据处理管道
</背景信息>

<指令>
- 编写干净、地道的 Python 代码
- 为函数签名包含类型提示
- 为公共函数添加文档字符串
- 遵循 PEP 8 风格指南
</指令>

<工具指导>
对 shell 操作使用 bash，对代码任务使用 python。
文件操作应该使用 pathlib 以实现跨平台兼容性。
</工具指导>

<输出描述>
提供带语法高亮的代码块。
在注释中解释不明显的决策。
</输出描述>
```

**示例 2：渐进文档加载**
```markdown
# 不要一次性加载所有文档：

# 步骤 1：加载摘要
docs/api_summary.md          # 轻量级概述

# 步骤 2：根据需要加载特定部分
docs/api/endpoints.md        # 仅在需要 API 调用时
docs/api/authentication.md   # 仅在需要认证上下文时
```

## 指南

1. 将上下文视为具有递减回报的有限资源
2. 将关键信息放在受青睐的注意力位置（开头和结尾）
3. 使用渐进式披露推迟加载直到需要
4. 用清晰的章节边界组织系统提示
5. 在开发期间监控上下文使用
6. 在 70-80% 利用率实现压缩触发器
7. 为上下文降级设计而不是希望避免它
8. 优先选择较小的高信号上下文而不是较大的低信号上下文

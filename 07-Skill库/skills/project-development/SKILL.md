# 项目开发方法论

此技能涵盖识别适合 LLM 处理的任务、设计有效的项目架构，以及使用智能体辅助开发进行快速迭代的原则。该方法论适用于构建批处理管道、多智能体研究系统或交互式智能体应用程序。

## 何时激活

在以下情况下激活此技能：
- 启动可能从 LLM 处理中受益的新项目
- 评估任务是否适合使用智能体而非传统代码
- 设计由 LLM 驱动的应用程序架构
- 规划具有结构化输出的批处理管道
- 在单智能体和多智能体方法之间进行选择
- 估算 LLM 密集型项目的成本和时间线

## 核心概念

### 任务-模型适配性识别

并非每个问题都能从 LLM 处理中受益。任何项目的第一步都是评估任务特征是否与 LLM 优势相符。此评估应在编写任何代码之前进行。

**适合 LLM 的任务具有以下特征：**

| 特征 | 为什么适合 |
|----------------|-------------|
| 跨来源综合 | LLM 擅长结合来自多个输入的信息 |
| 带有评分标准的主观判断 | LLM 处理评分、评估和基于标准的分类 |
| 自然语言输出 | 当目标是人类可读的文本，而非结构化数据时 |
| 容错性 | 个别失败不会破坏整个系统 |
| 批处理 | 项目之间不需要对话状态 |
| 训练中的领域知识 | 模型已经具有相关上下文 |

**不适合 LLM 的任务具有以下特征：**

| 特征 | 为什么失败 |
|----------------|--------------|
| 精确计算 | 数学、计数和精确算法不可靠 |
| 实时要求 | LLM 延迟对于亚秒级响应来说太高 |
| 完美准确度要求 | 幻觉风险使 100% 准确度成为不可能 |
| 依赖专有数据 | 模型缺乏必要的上下文 |
| 顺序依赖 | 每个步骤严重依赖前一个结果 |
| 确定性输出要求 | 相同输入必须产生相同输出 |

评估应通过手动原型设计进行：在构建任何自动化之前，选取一个有代表性的示例并用目标模型直接测试它。

### 手动原型步骤

在投入自动化之前，使用手动测试验证任务-模型适配性。将一个有代表性的输入复制到模型界面中。评估输出质量。这需要几分钟，但可以防止数小时的浪费开发。

此验证回答关键问题：
- 模型是否具有完成此任务所需的知识？
- 模型能否以你需要的格式产生输出？
- 你应该期望什么样的质量水平？
- 有哪些明显的失败模式需要解决？

如果手动原型失败，自动化系统也会失败。如果成功，你就有了比较的基线和提示设计的模板。

### 管道架构

LLM 项目受益于分阶段管道架构，其中每个阶段都是：
- **离散的**：阶段之间有清晰的边界
- **幂等的**：重新运行产生相同的结果
- **可缓存的**：中间结果持久化到磁盘
- **独立的**：每个阶段可以单独运行

**规范的管道结构：**

```
获取 → 准备 → 处理 → 解析 → 渲染
```

1. **获取**：从来源获取原始数据（API、文件、数据库）
2. **准备**：将数据转换为提示格式
3. **处理**：执行 LLM 调用（昂贵的非确定性步骤）
4. **解析**：从 LLM 输出中提取结构化数据
5. **渲染**：生成最终输出（报告、文件、可视化）

阶段 1、2、4 和 5 是确定性的。阶段 3 是非确定性和昂贵的。这种分离允许仅在必要时重新运行昂贵的 LLM 阶段，同时快速迭代解析和渲染。

### 文件系统作为状态机

使用文件系统来跟踪管道状态，而不是数据库或内存结构。每个处理单元获得一个目录。每个阶段完成由文件存在标记。

```
data/{id}/
├── raw.json         # 获取阶段完成
├── prompt.md        # 准备阶段完成
├── response.md      # 处理阶段完成
├── parsed.json      # 解析阶段完成
```

检查项目是否需要处理：检查输出文件是否存在。要重新运行阶段：删除其输出文件和下游文件。要调试：直接读取中间文件。

此模式提供：
- 自然幂等性（文件存在控制执行）
- 易于调试（所有状态都是人类可读的）
- 简单并行化（每个目录是独立的）
- 简单的缓存（文件在运行之间持久化）

### 结构化输出设计

当 LLM 输出必须以编程方式解析时，提示设计直接决定了解析的可靠性。提示必须指定确切的格式要求并附带示例。

**有效的结构规范包括：**

1. **章节标记**：用于解析的显式标题或前缀
2. **格式示例**：准确展示输出应该是什么样子
3. **原理说明**："我将以编程方式解析此内容"
4. **约束值**：枚举选项、分数范围、格式

**示例提示结构：**
```
分析以下内容，并以完全此格式提供你的响应：

## 摘要
[你的摘要放在这里]

## 分数
评分：[1-10]

## 详情
- 要点 1
- 要点 2

完全遵循此格式，因为我将以编程方式解析它。
```

解析代码必须优雅地处理变化。LLM 并不能完美地遵循指令。构建能够：
- 使用足够灵活的正则表达式模式来处理轻微的格式变化
- 当章节缺失时提供合理的默认值
- 记录解析失败以供后续审查，而不是崩溃

### 智能体辅助开发

现代具备智能体能力的模型可以显著加速开发。模式是：

1. 描述项目目标和约束
2. 让智能体生成初始实现
3. 针对特定失败进行测试和迭代
4. 根据结果优化提示和架构

这是关于快速迭代：生成、测试、修复、重复。智能体处理样板代码和初始结构，而你专注于领域特定需求和边缘情况。

有效智能体辅助开发的关键实践：
- 预先提供清晰、具体的需求
- 将大型项目分解为离散的组件
- 在继续下一个之前测试每个组件
- 让智能体一次专注于一个任务

### 成本和规模估算

LLM 处理具有可预测的成本，应在开始之前进行估算。公式：

```
总成本 = (项目数 × 每项令牌数 × 每令牌价格) + API 开销
```

对于批处理：
- 估算每项的输入令牌数（提示 + 上下文）
- 估算每项的输出令牌数（典型响应长度）
- 乘以项目数量
- 添加 20-30% 的缓冲用于重试和失败

在开发过程中跟踪实际成本。如果成本显著超出估算，重新评估方法。考虑：
- 通过截断减少上下文长度
- 对更简单的项目使用较小的模型
- 缓存和重用部分结果
- 并行处理以减少实际时间（而非令牌成本）

## 详细主题

### 选择单智能体 vs 多智能体架构

单智能体管道适用于：
- 具有独立项目的批处理
- 项目之间不交互的任务
- 更简单的成本和复杂性管理

多智能体架构适用于：
- 并行探索不同方面
- 超出单一上下文窗口容量的任务
- 当专门的子智能体提高质量时

多智能体的主要原因是上下文隔离，而非角色拟人化。子智能体获得新的上下文窗口以专注于子任务。这可以防止长时间运行任务上的上下文退化。

参见 `multi-agent-patterns` 技能获取详细的架构指导。

### 架构简化

从最小架构开始。仅在证明必要时添加复杂性。生产证据表明，移除专用工具通常会提高性能。

Vercel 的 d0 智能体通过将 17 个专用工具减少到 2 个原语：bash 命令执行和 SQL，实现了 100% 的成功率（从 80% 提升）。文件系统智能体模式使用标准 Unix 工具（grep、cat、find、ls）而非自定义探索工具。

**何时简化胜过复杂性：**
- 你的数据层文档完善且结构一致
- 模型具有足够的推理能力
- 你的专用工具限制而非赋能模型
- 你花费更多时间维护脚手架而非改善结果

**何时复杂性是必要的：**
- 你的底层数据混乱、不一致或文档不完善
- 领域需要模型缺乏的专门知识
- 安全约束要求限制智能体能力
- 操作真正复杂且受益于结构化工作流

参见 `tool-design` 技能获取详细的工具架构指导。

### 迭代和重构

期望重构。大规模生产智能体系统需要多次架构迭代。Manus 自推出以来重构了他们的智能体框架五次。Bitter Lesson 表明，为当前模型限制添加的结构会随着模型改进而成为约束。

为变化而构建：
- 保持架构简单且不带偏见
- 跨模型优势测试以验证你的框架没有限制性能
- 设计从模型改进中受益的系统，而非锁定限制

## 实用指导

### 项目规划模板

1. **任务分析**
   - 输入是什么？期望的输出是什么？
   - 这是综合、生成、分类还是分析？
   - 什么样的错误率是可接受的？
   - 每次成功完成的价值是什么？

2. **手动验证**
   - 用目标模型测试一个示例
   - 评估输出质量和格式
   - 识别失败模式
   - 估算每项的令牌数

3. **架构选择**
   - 单管道 vs 多智能体
   - 所需的工具和数据源
   - 存储和缓存策略
   - 并行化方法

4. **成本估算**
   - 项目数 × 令牌数 × 价格
   - 开发时间
   - 基础设施需求
   - 持续运营成本

5. **开发计划**
   - 分阶段实施
   - 每阶段的测试策略
   - 迭代里程碑
   - 部署方法

### 避免的反模式

**跳过手动验证**：在验证模型能否完成任务之前构建自动化，当方法存在根本缺陷时会浪费大量时间。

**单体管道**：将所有阶段合并到一个脚本中会使调试和迭代变得困难。用持久化的中间输出来分离阶段。

**过度约束模型**：添加护栏、预过滤和验证逻辑，而模型可以自己处理这些。测试你的脚手架是帮助还是伤害。

**忽视成本直到生产**：令牌成本在大规模下迅速累积。从一开始就进行估算和跟踪。

**完美的解析要求**：期望 LLM 完美地遵循格式指令。构建能够处理变化的健壮解析器。

**过早优化**：在基本管道正常工作之前添加缓存、并行化和优化。

## 示例

**示例 1：批分析管道（Karpathy 的 HN 时间胶囊）**

任务：用事后评分分析 10 年前的 930 个 HN 讨论。

架构：
- 5 阶段管道：获取 → 提示 → 分析 → 解析 → 渲染
- 文件系统状态：data/{date}/{item_id}/ 包含阶段输出文件
- 结构化输出：6 个章节，具有明确的格式要求
- 并行执行：15 个 LLM 调用工作器

结果：总成本 $58，执行约 1 小时，静态 HTML 输出。

**示例 2：架构简化（Vercel d0）**

任务：用于内部分析的文本到 SQL 智能体。

之前：17 个专用工具，80% 成功率，平均执行 274 秒。

之后：2 个工具（bash + SQL），100% 成功率，平均执行 77 秒。

关键洞察：语义层已经是良好的文档。Claude 只需要访问直接读取文件。

参见[案例研究](./references/case-studies.md)获取详细分析。

## 指导原则

1. 在构建自动化之前，使用手动原型验证任务-模型适配性
2. 将管道结构化为离散的、幂等的、可缓存的阶段
3. 使用文件系统进行状态管理和调试
4. 设计提示以生成结构化、可解析的输出，并附带明确的格式示例
5. 从最小架构开始；仅在证明必要时添加复杂性
6. 尽早估算成本并在整个开发过程中跟踪
7. 构建能够处理 LLM 输出变化的健壮解析器
8. 期望并计划进行多次架构迭代
9. 测试脚手架是帮助还是限制模型性能
10. 使用智能体辅助开发进行实现的快速迭代

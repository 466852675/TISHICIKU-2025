# 案例研究：LLM 项目开发

本参考包含生产级 LLM 项目的详细案例研究，展示有效的开发方法论。每个案例研究分析问题、方法、架构和经验教训。

---

## 案例研究 1：Karpathy 的 HN 时间胶囊

**来源**：https://github.com/karpathy/hn-time-capsule

### 问题陈述

分析 10 年前的 Hacker News 讨论，并根据事后之明对评论者的预测进行评分。

### 任务-模型适配分析

此任务非常适合 LLM 处理，因为：

| 因素 | 评估 |
|------|------|
| 综合 | 结合文章内容 + 多个评论线程 |
| 主观判断 | 根据已知结果对预测进行评分 |
| 领域知识 | 模型了解实际发生的事情 |
| 错误容忍度 | 一个评论的错误评分不会破坏系统 |
| 批处理 | 每篇文章都是独立的 |
| 自然语言输出 | 人类可读的分析是目标 |

### 开发方法论

**步骤 1：手动原型**

在构建任何自动化之前，Karpathy 将一篇文章 + 评论线程复制粘贴到 ChatGPT 中以验证方法。这只需几分钟，并确认了：
- 模型可以产生有洞察力的事后分析
- 输出格式适用于预期用例
- 质量超过他手动能做的

**步骤 2：代理辅助实现**

使用 Opus 4.5 在大约 3 小时内构建流水线。代理处理了：
- HN 首页的 HTML 解析
- 用于评论的 Algolia API 集成
- 提示模板设计
- 输出解析逻辑
- 静态 HTML 渲染

**步骤 3：批处理执行**

- 930 次 LLM 查询（31 天 × 30 篇文章）
- 15 个并行工作线程
- ~58 美元总成本
- ~1 小时执行时间

### 流水线架构

```
fetch → prompt → analyze → parse → render
```

**阶段 1：Fetch**
- 下载目标日期的 HN 首页
- 通过 HTTP 获取文章内容
- 通过 Algolia API 获取评论
- 输出：`data/{date}/{item_id}/meta.json`, `article.txt`, `comments.json`

**阶段 2：Prompt**
- 加载文章元数据和内容
- 加载评论树
- 从模板生成 markdown 提示
- 输出：`data/{date}/{item_id}/prompt.md`

**阶段 3：Analyze**
- 提交提示到 GPT 5.1 Thinking API
- 使用 ThreadPoolExecutor 并行执行
- 输出：`data/{date}/{item_id}/response.md`

**阶段 4：Parse**
- 通过正则从 "Final grades" 章节提取评分
- 通过正则提取有趣度评分
- 汇总所有文章的评分
- 输出：`data/{date}/{item_id}/grades.json`, `score.json`

**阶段 5：Render**
- 生成带嵌入式 JavaScript 的静态 HTML
- 创建带文章导航的每日页面
- 创建带汇总排名的名人堂
- 输出：`output/{date}/index.html`, `output/hall-of-fame.html`

### 结构化输出设计

提示模板指定确切的输出格式：

```
让我们利用现在的事后之明分 6 个部分：

1. 简要总结文章和讨论线程。
2. 这个话题最终发生了什么？
3. 颁发"最有先见之明"和"最错误"评论奖。
4. 提及任何其他有趣或值得注意的方面。
5. 对特定人员的评论进行评分。
6. 最后，给出最终评分（0-10）。

关于第 5 部分的格式，使用标题 "Final grades"，后面跟着
一个简单的无序列表，格式为 "name: grade (optional comment)"。

请严格遵循格式，因为我将通过编程方式解析它。
```

关键技术：
- 编号章节用于结构
- 带示例的显式格式规范
- 理由披露（"因为我将通过编程方式解析它"）
- 约束输出（字母评分、0-10 分）

### 解析实现

解析代码优雅地处理变化：

```python
def parse_grades(text: str) -> dict[str, dict]:
    # 匹配 "Final grades"，可选章节号或 markdown
    pattern = r'(?:^|\n)(?:\d+[\.\)]\s*)?(?:#+ *)?Final grades\s*\n'
    match = re.search(pattern, text, re.IGNORECASE)
    
    # 处理 ASCII 和 Unicode 减号
    line_pattern = r'^[\-\*]\s*([^:]+):\s*([A-F][+\-\−]?)(?:\s*\(([^)]+)\))?'
```

### 经验教训

1. **首先手动验证**：5 分钟的复制粘贴测试防止了数小时的浪费开发。

2. **文件系统作为状态**：每篇文章目录包含所有中间输出，使调试变得简单。

3. **幂等阶段**：重新运行仅处理缺少输出文件的项目。

4. **代理辅助开发**：通过专注于需求而非实现细节，3 小时获得工作代码。

5. **并行执行**：15 个工作线程减少了执行时间，而没有增加 token 成本。

## 案例研究 2：Vercel d0 架构简化

**来源**：https://vercel.com/blog/we-removed-80-percent-of-our-agents-tools

### 问题陈述

构建一个 text-to-SQL 代理，使 Vercel 的任何人都能通过 Slack 中的自然语言问题查询分析数据。

### 初始方法（失败）

团队构建了一个复杂的系统，包括：
- 17 个专业工具（模式查找、查询验证、错误恢复等）
- 约束推理的大量提示工程
- 仔细的上下文管理
- 模式信息的手编码检索

**结果**：
- 80% 成功率
- 274.8 秒平均执行时间
- ~102k token 平均使用
- ~12 步平均
- 持续的维护负担

### 问题

团队正在解决模型自己能处理的问题：
- 预过滤上下文
- 约束选项
- 将每次交互包装在验证逻辑中
- 构建工具来"保护"模型免受复杂性

每个边缘情况都需要另一个补丁。每次模型更新都需要重新校准约束。花在维护脚手架上的时间比改进结果还多。

### 架构简化

假设：如果我们只是给 Claude 访问原始文件的权限，让它自己解决会怎样？

**新架构**：
- 总共 2 个工具：ExecuteCommand (bash) + ExecuteSQL
- 通过沙箱直接文件系统访问
- 语义层作为 YAML/Markdown/JSON 文件
- 标准 Unix 工具（grep、cat、find、ls）

```javascript
const agent = new ToolLoopAgent({
  model: "anthropic/claude-opus-4.5",
  tools: {
    ExecuteCommand: executeCommandTool(sandbox),
    ExecuteSQL,
  },
});
```

### 结果

| 指标 | 之前（17 个工具） | 之后（2 个工具） | 变化 |
|------|-------------------|-----------------|------|
| 平均执行时间 | 274.8s | 77.4s | 3.5 倍更快 |
| 成功率 | 80% | 100% | +20% |
| 平均 token 使用 | ~102k | ~61k | 减少 37% |
| 平均步数 | ~12 | ~7 | 减少 42% |

之前的最坏情况：724 秒、100 步、145k token，仍然失败。
之后的相同查询：141 秒、19 步、67k token，成功。

### 为什么有效

1. **良好的文档已经存在**：语义层文件包含维度定义、度量计算和连接关系。工具正在总结已经可读的内容。

2. **文件系统是成熟的抽象**：模型从训练中深刻理解文件系统。grep 已有 50 年历史，运行完美。

3. **约束变成了负担**：有了更好的模型，护栏限制性能的程度超过了帮助。

### 关键教训

1. **减法加法**：最好的代理可能是工具最少的代理。每个工具都是你在为模型做出的选择。

2. **为未来模型构建**：模型改进速度比工具快。为今天优化的架构可能对明天过度约束。

3. **良好的上下文胜过巧妙的工具**：投资于文档、清晰的命名和结构良好的数据。这个基础比复杂的工具更重要。

4. **从简单开始**：模型 + 文件系统 + 目标。仅在证明必要时添加复杂性。

## 案例研究 3：Manus 上下文工程

**来源**：Peak Ji 的博客"AI 代理的上下文工程：构建 Manus 的经验教训"

### 问题陈述

构建一个通用消费代理，可以在维护性能和成本的同时完成跨越 50+ 工具调用的复杂任务。

### 核心洞察

KV 缓存命中率是生产代理的最重要指标。它直接影响延迟和成本。

- Claude Sonnet 缓存：$0.30/MTok
- Claude Sonnet 未缓存：$3.00/MTok
- 10 倍成本差异

在代理工作负载中平均 100:1 的输入输出比下，优化缓存命中率主导了成本方程。

### 关键模式

**1. 仅追加上下文**

永远不要修改先前的操作或观察。确保确定性序列化（JSON 键顺序必须稳定）。单个 token 差异会使该点之后的缓存失效。

常见错误：在系统提示开头包含时间戳会完全扼杀缓存命中率。

**2. 掩码，不要移除**

不要在迭代中动态添加或移除工具。工具定义位于上下文的前端——任何更改都会使所有后续内容的 KV 缓存失效。

相反，在解码期间使用 logit 掩码来约束工具选择，而不修改定义。这在仍然控制行为的同时保持缓存。

**3. 文件系统作为上下文**

将文件系统视为无限的、持久的、代理可操作的记忆。模型学会按需写入和读取文件。

压缩策略应该是可恢复的：
- 如果保留了 URL，可以丢弃网页内容
- 如果文件路径仍然可用，可以省略文档内容

**4. 背诵以集中注意力**

Manus 创建一个 todo.md 文件并逐步更新。这不仅仅是组织——它将全局计划推入模型的近期注意力范围。

通过在上下文末尾不断重写目标，代理避免了"迷失在中间"的问题并保持目标一致性。

**5. 在上下文中保留错误**

不要隐藏失败。当模型看到失败的操作和由此产生的错误时，它会隐式更新信念并避免重复错误。

擦除失败会移除模型适应所需的证据。

### 用于上下文隔离的多代理

Manus 中子代理的主要目标是上下文隔离，而不是角色划分。对于需要离散工作的任务：
- 规划器将任务分配给具有自己上下文窗口的子代理
- 简单任务：通过函数调用传递指令
- 复杂任务：与子代理共享完整上下文

子代理有一个具有约束输出模式的 submit_results 工具。约束解码确保遵守定义的格式。

### 分层动作空间

与其将每个工具绑定为工具：
- 少量（<20）原子函数：Bash、文件系统访问、代码执行
- 大多数动作卸载到沙箱层
- 通过 CLI 暴露的 MCP 工具，通过 Bash 工具执行

这减少了工具定义 token 并防止模型因描述重叠而混淆。

### 迭代期望

Manus 自推出以来已经重构了他们的代理框架五次。苦涩的教训表明，为当前限制添加的结构会随着模型改进而成为约束。

跨模型优势测试以验证你的工具没有限制性能。简单、无偏见的设计能更好地适应模型改进。

## 案例研究 4：Anthropic 多代理研究

**来源**：Anthropic 博客"我们如何构建多代理研究系统"

### 问题陈述

构建一个研究功能，可以使用多个并行代理探索复杂主题，跨 Web、Google Workspace 和集成进行搜索。

### 架构

协调器-工作者模式：
- 主导代理分析查询并制定策略
- 主导生成子代理进行并行探索
- 子代理将发现返回给主导进行综合
- 引用代理处理最终输出

### 性能洞察

三个因素解释了 BrowseComp 评估中 95% 的性能差异：
- Token 使用：80% 的方差
- 工具调用次数：额外因素
- 模型选择：额外因素

多代理架构有效地扩展了 token 使用，以完成超出单代理限制的任务。

### Token 经济学

- 聊天交互：基线
- 单代理：比聊天多 ~4 倍 token
- 多代理：比聊天多 ~15 倍 token

多代理需要高价值任务来证明成本。

### 提示原则

1. **像你的代理一样思考**：构建模拟，逐步观察，识别失败模式。

2. **教授委托**：子代理需要目标、输出格式、工具/来源指导以及清晰的边界。

3. **将努力与复杂性相匹配**：按任务类型明确指导代理/工具调用次数。

4. **工具设计至关重要**：每个工具都有独特的目的和清晰的描述。糟糕的描述会完全将代理引向错误的方向。

5. **让代理自我改进**：Claude 4 模型可以诊断提示失败并建议改进。工具测试代理可以重写工具描述以避免常见错误。

6. **先广泛，后深入**：首先进行广泛查询，评估全景，然后深入 specifics。

7. **引导思考过程**：扩展思考模式作为规划的可控草稿本。

8. **并行工具调用**：3-5 个并行子代理，每个子代理并行调用 3+ 个工具。将研究时间减少多达 90%。

### 评估方法

- 立即从 ~20 个代表性查询开始
- 带评分标准的 LLM 作为评判：事实准确性、引用准确性、完整性、来源质量、工具效率
- 人工评估捕捉自动化遗漏的边缘情况
- 专注于多轮代理的端到端评估

## 跨案例模式

### 共同成功因素

1. **自动化前手动验证**：所有成功的项目首先通过简单测试验证了任务-模型适配。

2. **文件系统作为基础**：无论是用于状态管理（Karpathy）、工具接口（Vercel）还是记忆（Manus），文件系统都提供了成熟的抽象。

3. **架构简单性**：在多个案例中，简化优于复杂性。从最小开始，仅添加证明必要的内容。

4. **结构化输出与稳健解析**：显式格式规范与处理变化的灵活解析相结合。

5. **迭代期望**：没有一个项目在第一次就得到正确的架构。为变化而构建。

### 共同失败模式

1. **过度约束模型**：帮助较弱模型的护栏会随着能力改进而成为负担。

2. **工具激增**：更多工具通常意味着更多混淆和更差的性能。

3. **隐藏错误**：从上下文中移除失败会阻止模型学习。

4. **过早优化**：在基本功能正常工作之前添加复杂性。

5. **忽视经济学**：Token 成本迅速累积；估算和跟踪至关重要。

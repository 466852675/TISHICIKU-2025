## 审查清单

**代码质量：**
- 关注点清晰分离？
- 适当的错误处理？
- 类型安全（如适用）？
- 遵循DRY原则？
- 处理边缘情况？

**架构：**
- 合理的设计决策？
- 可扩展性考虑？
- 性能影响？
- 安全问题？

**测试：**
- 测试实际测试逻辑（而非模拟）？
- 覆盖边缘情况？
- 需要时进行集成测试？
- 所有测试通过？

**要求：**
- 满足所有计划要求？
- 实施符合规范？
- 无范围蔓延？
- 记录重大更改？

**生产准备：**
- 迁移策略（如果模式更改）？
- 考虑向后兼容性？
- 文档完整？
- 无明显错误？

## 关键规则

**要：**
- 按实际严重性分类（并非一切都是严重）
- 具体（文件:行号，而非模糊）
- 解释为什么问题重要
- 承认优势
- 给出明确裁决

**不要：**
- 未检查就说"看起来不错"
- 将吹毛求疵标记为严重
- 对您未审查的代码给出反馈
- 模糊（"改进错误处理"）
- 避免给出明确裁决

# 上下文压缩评估框架

本文档提供测量上下文压缩质量的完整评估框架，包括探针类型、评分标准和 LLM 评判者配置。

## 探针类型

### 回忆探针

测试对对话历史中特定细节的事实保留。

**结构：**
```
问题：[询问截断历史中的具体事实]
预期：[应该保留的确切细节]
评分：技术细节的匹配准确性
```

**示例：**
- "开始这次调试会话的原始错误消息是什么？"
- "我们决定使用哪个版本的依赖项？"
- "失败的确切命令是什么？"

### 工件探针

测试文件跟踪和修改意识。

**结构：**
```
问题：[询问创建、修改或检查的文件]
预期：[带变更描述的完整列表]
评分：文件列表的完整性和变更描述的准确性
```

**示例：**
- "我们修改了哪些文件？描述每个文件的变化。"
- "这次会话中我们创建了什么新文件？"
- "我们检查了哪些配置文件但没有更改？"

### 延续探针

测试无需重新获取上下文即可继续工作的能力。

**结构：**
```
问题：[询问下一步或当前状态]
预期：[基于会话历史的可操作下一步]
评分：无需请求重新读取文件即可继续的能力
```

**示例：**
- "接下来我们应该做什么？"
- "哪些测试仍然失败，为什么？"
- "上一步留下了什么未完成的内容？"

### 决策探针

测试推理链和决策理由的保留。

**结构：**
```
问题：[询问为什么做出某个决定]
预期：[导致该决定的推理]
评分：决策背景和考虑的替代方案的保留
```

**示例：**
- "我们讨论了 Redis 问题的选项。我们决定了什么，为什么？"
- "为什么我们选择连接池而不是每个请求的连接？"
- "对于认证修复，我们考虑了哪些替代方案？"

## 评分标准

### 准确性维度

| 标准 | 问题 | 分数 0 | 分数 3 | 分数 5 |
|-----------|----------|---------|---------|---------|
| accuracy_factual | 事实、文件路径和技术细节是否正确？ | 完全错误或虚构 | 大部分准确，有轻微错误 | 完全准确 |
| accuracy_technical | 代码引用和技术概念是否正确？ | 重大技术错误 | 基本正确，有轻微问题 | 技术上精确 |

### 上下文意识维度

| 标准 | 问题 | 分数 0 | 分数 3 | 分数 5 |
|-----------|----------|---------|---------|---------|
| context_conversation_state | 响应是否反映当前对话状态？ | 对先前上下文没有意识 | 有一般意识，但有差距 | 完全了解对话历史 |
| context_artifact_state | 响应是否反映了访问了哪些文件/工件？ | 对工件没有意识 | 部分工件意识 | 完整的工件状态意识 |

### 工件轨迹维度

| 标准 | 问题 | 分数 0 | 分数 3 | 分数 5 |
|-----------|----------|---------|---------|---------|
| artifact_files_created | 代理是否知道创建了哪些文件？ | 没有知识 | 知道大多数文件 | 完全了解 |
| artifact_files_modified | 代理是否知道修改了哪些文件以及更改了什么？ | 没有知识 | 对大多数修改有很好的了解 | 对所有修改完全了解 |
| artifact_key_details | 代理是否记得函数名、变量名、错误消息？ | 没有回忆 | 回忆起大多数关键细节 | 完全回忆 |

### 完整性维度

| 标准 | 问题 | 分数 0 | 分数 3 | 分数 5 |
|-----------|----------|---------|---------|---------|
| completeness_coverage | 响应是否涵盖了问题的所有部分？ | 忽略了大多数部分 | 涵盖了大多数部分 | 彻底涵盖了所有部分 |
| completeness_depth | 是否提供了足够的细节？ | 表面或缺少细节 | 足够的细节 | 全面的细节 |

### 连续性维度

| 标准 | 问题 | 分数 0 | 分数 3 | 分数 5 |
|-----------|----------|---------|---------|---------|
| continuity_work_state | 代理能否在不重新获取先前访问的信息的情况下继续？ | 无法在不重新获取所有上下文的情况下继续 | 可以在最小重新获取的情况下继续 | 可以无缝继续 |
| continuity_todo_state | 代理是否保持对待处理任务的意识？ | 失去了对所有待办事项跟踪 | 有很好的意识，但有一些差距 | 完美的任务意识 |
| continuity_reasoning | 代理是否保留先前决策背后的理由？ | 不记得推理 | 一般记得推理 | 优秀的保留 |

### 指令遵循维度

| 标准 | 问题 | 分数 0 | 分数 3 | 分数 5 |
|-----------|----------|---------|---------|---------|
| instruction_format | 响应是否遵循请求的格式？ | 忽略格式 | 一般遵循格式 | 完全遵循格式 |
| instruction_constraints | 响应是否尊重所述约束？ | 忽略约束 | 大部分尊重约束 | 完全尊重所有约束 |

## LLM 评判者配置

### 系统提示

```
您是一位专家评估员，评估软件开发对话中的 AI 助手响应。

您的任务是根据特定标准对响应进行评分。对于每个标准：
1. 仔细阅读标准问题
2. 检查响应中的证据
3. 根据评分指南分配 0-5 的分数
4. 为您的分数提供简要理由

保持客观和一致。关注响应中存在的内容，而不是可能包含的内容。
```

### 评判者输入格式

```json
{
  "probe_question": "原始错误消息是什么？",
  "model_response": "[要评估的响应]",
  "compacted_context": "[提供的压缩上下文]",
  "ground_truth": "[可选：已知正确答案]",
  "rubric_criteria": ["accuracy_factual", "accuracy_technical", "context_conversation_state"]
}
```

### 评判者输出格式

```json
{
  "criterionResults": [
    {
      "criterionId": "accuracy_factual",
      "score": 5,
      "reasoning": "响应正确识别了 401 错误、特定端点和根本原因。"
    }
  ],
  "aggregateScore": 4.8,
  "dimensionScores": {
    "accuracy": 4.9,
    "context_awareness": 4.5,
    "artifact_trail": 3.2,
    "completeness": 5.0,
    "continuity": 4.8,
    "instruction_following": 5.0
  }
}
```

## 基准结果参考

跨压缩方法的性能（基于 36,000+ 条消息）：

| 方法 | 总体 | 准确性 | 上下文 | 工件 | 完整性 | 连续性 | 指令遵循 |
|--------|---------|----------|---------|----------|----------|------------|-------------|
| 锚定迭代 | 3.70 | 4.04 | 4.01 | 2.45 | 4.44 | 3.80 | 4.99 |
| 再生 | 3.44 | 3.74 | 3.56 | 2.33 | 4.37 | 3.67 | 4.95 |
| 不透明 | 3.35 | 3.43 | 3.64 | 2.19 | 4.37 | 3.77 | 4.92 |

**关键发现：**

1. **准确性差距**：最佳和最差方法之间相差 0.61 分
2. **上下文意识差距**：相差 0.45 分，锚定迭代占优
3. **工件轨迹**：普遍较弱（2.19-2.45），需要专门处理
4. **完整性和指令遵循**：差异最小

## 统计考虑

- 0.26-0.35 分的差异在任务类型和会话长度上保持一致
- 模式对短会话和长会话都适用
- 模式在调试、功能实现和代码审查任务中都适用
- 样本量：数百个压缩点的 36,611 条消息

## 实现说明

### 探针生成

基于截断历史在每个压缩点生成探针：
1. 提取事实声明用于回忆探针
2. 提取文件操作用于工件探针
3. 提取未完成的任务用于延续探针
4. 提取决策点用于决策探针

### 评分过程

1. 将探针问题 + 模型响应 + 压缩上下文提供给评判者
2. 根据标准中的每个标准进行评估
3. 输出带有分数和理由的结构化 JSON
4. 将维度分数计算为加权平均值
5. 将总体分数计算为维度的未加权平均值

### 盲法

评判者不应该知道哪个压缩方法产生了正在评估的响应。这可以防止对已知方法的偏见。

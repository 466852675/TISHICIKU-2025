# 智能体工具设计

工具是智能体与世界交互的主要机制。它们定义了确定性系统与非确定性智能体之间的契约。与为开发者设计的传统软件 API 不同，工具 API 必须专为语言模型设计，使其能够推理意图、推断参数值，并从自然语言请求生成调用。糟糕的工具设计会产生任何提示工程都无法修复的失败模式。有效的工具设计遵循特定原则，考虑智能体如何感知和使用工具。

## 何时激活

在以下情况下激活此技能：
- 为智能体系统创建新工具
- 调试工具相关的失败或误用
- 优化现有工具集以获得更好的智能体性能
- 从头设计工具 API
- 评估第三方工具以集成到智能体
- 在整个代码库中标准化工具规范

## 核心概念

工具是确定性系统与非确定性智能体之间的契约。整合原则指出，如果人类工程师无法明确说明在特定情况下应该使用哪个工具，就不能期望智能体能做得更好。有效的工具描述是塑造智能体行为的提示工程。

关键原则包括：清晰的描述回答做什么、何时做和返回什么；在完整性和令牌效率之间平衡的响应格式；支持恢复的错误消息；以及减少认知负担的一致规范。

## 详细主题

### 智能体-工具接口

**工具作为契约**
工具是确定性系统与非确定性智能体之间的契约。当人类调用 API 时，他们理解契约并做出适当的请求。智能体必须从描述中推断契约，并生成匹配预期格式的调用。

这种根本差异需要重新思考 API 设计。契约必须明确无误，示例必须说明预期模式，错误消息必须指导纠正。工具定义中的每个模糊之处都成为潜在的失败模式。

**工具描述作为提示**
工具描述被加载到智能体上下文中，共同引导行为。描述不仅仅是文档——它们是塑造智能体如何推理工具使用的提示工程。

糟糕的描述如"搜索数据库"加上神秘的参数名称会迫使智能体猜测。优化的描述包括使用上下文、示例和默认值。描述回答：工具做什么、何时使用它以及它产生什么。

**命名空间和组织**
随着工具集合的增长，组织变得至关重要。命名空间将相关工具分组在共同前缀下，帮助智能体在正确的时间选择适当的工具。

命名空间在功能之间创建清晰的边界。当智能体需要数据库信息时，它路由到数据库命名空间。当它需要网络搜索时，它路由到网络命名空间。

### 整合原则

**单一综合工具**
整合原则指出，如果人类工程师无法明确说明在特定情况下应该使用哪个工具，就不能期望智能体能做得更好。这导致偏好单一综合工具而非多个狭窄工具。

不要实现 list_users、list_events 和 create_event，而是实现查找可用性并安排日程的 schedule_event。综合工具在内部处理完整的工作流，而不是要求智能体链式调用多个函数。

**为什么整合有效**
智能体的上下文和注意力有限。集合中的每个工具都在工具选择阶段竞争注意力。每个工具都添加消耗上下文预算的描述令牌。重叠的功能造成应该使用哪个工具的模糊性。

整合通过消除冗余描述来减少令牌消耗。它通过让一个工具覆盖每个工作流来消除模糊性。它通过缩小有效工具集来减少工具选择复杂性。

**何时不整合**
整合并非普遍正确。具有根本不同行为的工具应保持分离。在不同上下文中使用的工具受益于分离。可能被独立调用的工具不应被人为捆绑。

### 架构简化

将整合原则推向其逻辑极端，导致架构简化：移除大多数专用工具，转而使用原始的通用能力。生产证据表明，这种方法可以胜过复杂的多工具架构。

**文件系统智能体模式**
不要为数据探索、模式查找和查询验证构建自定义工具，而是通过单一命令执行工具提供直接文件系统访问。智能体使用标准 Unix 工具（grep、cat、find、ls）来探索、理解和操作系统。

这之所以有效，是因为：
1. 文件系统是一种经过验证的抽象，模型深刻理解
2. 标准工具具有可预测的、文档完善的行为
3. 智能体可以灵活地链式原始工具，而不是受限于预定义的工作流
4. 文件中的良好文档取代了汇总工具的需求

**何时简化胜过复杂性**
简化在以下情况下有效：
- 你的数据层文档完善且结构一致
- 模型具有足够的推理能力来导航复杂性
- 你的专用工具限制而非赋能模型
- 你花费更多时间维护脚手架而非改善结果

简化在以下情况下失败：
- 你的底层数据混乱、不一致或文档不完善
- 领域需要模型缺乏的专门知识
- 安全约束要求限制智能体能做什么
- 操作真正复杂且受益于结构化工作流

**停止限制推理**
一个常见的反模式是构建工具来"保护"模型免受复杂性。预过滤上下文、限制选项、在验证逻辑中包装交互。这些护栏随着模型改进往往成为负担。

要问的问题：你的工具是在启用新能力，还是在限制模型可以自己处理的推理？

**为未来模型构建**
模型改进速度超过工具跟上。为当今模型优化的架构可能对明天的模型来说过于受限。构建最小架构，可以从模型改进中受益，而不是锁定当前限制的复杂架构。

参见[架构简化案例研究](./references/architectural_reduction.md)了解生产证据。

### 工具描述工程

**描述结构**
有效的工具描述回答四个问题：

工具做什么？清晰、具体的功能描述。避免模糊的语言如"帮助"或"可以用于"。准确说明工具完成什么。

何时应该使用它？具体的触发器和上下文。包括直接触发器（"用户询问定价"）和间接信号（"需要当前市场费率"）。

它接受什么输入？带有类型、约束和默认值的参数描述。解释每个参数控制什么。

它返回什么？输出格式和结构。包括成功响应和错误条件的示例。

**默认参数选择**
默认值应反映常见用例。它们通过消除不必要的参数规范来减少智能体负担。它们防止因省略参数而导致的错误。

### 响应格式优化

工具响应大小显著影响上下文使用。实现响应格式选项让智能体控制详细程度。

简洁格式仅返回基本字段，适用于确认或基本信息。详细格式返回包含所有字段的完整对象，适用于需要完整上下文进行决策时。

在工具描述中包括关于何时使用每种格式的指导。智能体学会根据任务要求选择适当的格式。

### 错误消息设计

错误消息服务于两个受众：调试问题的开发人员和从失败中恢复的智能体。对于智能体，错误消息必须是可操作的。它们必须告诉智能体出了什么问题以及如何纠正。

设计支持恢复的错误消息。对于可重试错误，包括重试指导。对于输入错误，包括纠正后的格式。对于缺失数据，包括需要什么。

### 工具定义模式

在所有工具中使用一致的模式。建立命名规范：工具名称使用动词-名词模式，跨工具一致的参数名称，一致的返回字段名称。

### 工具集合设计

研究表明工具描述重叠会导致模型混淆。更多工具并不总是带来更好的结果。一个合理的指导方针是大多数应用使用 10-20 个工具。如果需要更多，使用命名空间创建逻辑分组。

实现帮助智能体选择正确工具的机制：工具分组、基于示例的选择，以及使用将路由到专用子工具的伞形工具的层次结构。

### MCP 工具命名要求

使用 MCP（模型上下文协议）工具时，始终使用完全限定工具名称以避免"找不到工具"错误。

格式：`ServerName:tool_name`

```python
# 正确：完全限定名称
"使用 BigQuery:bigquery_schema 工具检索表模式。"
"使用 GitHub:create_issue 工具创建问题。"

# 错误：非限定名称
"使用 bigquery_schema 工具..."  # 当有多个服务器时可能失败
```

没有服务器前缀，智能体可能无法定位工具，特别是当多个 MCP 服务器可用时。建立在所有工具引用中包括服务器上下文的命名规范。

### 使用智能体优化工具

Claude 可以优化自己的工具。当给定工具和观察到的失败模式时，它会诊断问题并提出改进建议。生产测试表明，这种方法通过帮助未来智能体避免错误，实现了任务完成时间减少 40%。

**工具测试智能体模式**：

```python
def optimize_tool_description(tool_spec, failure_examples):
    """
    使用智能体分析工具失败并改进描述。
    
    流程：
    1. 智能体尝试在多样化任务中使用工具
    2. 收集失败模式和摩擦点
    3. 智能体分析失败并提出改进建议
    4. 针对相同任务测试改进后的描述
    """
    prompt = f"""
    分析此工具规范和观察到的失败。
    
    工具：{tool_spec}
    
    观察到的失败：
    {failure_examples}
    
    识别：
    1. 智能体为何在此工具上失败
    2. 描述中缺少什么信息
    3. 什么模糊性导致错误使用
    
    提出解决这些问题的改进工具描述。
    """
    
    return get_agent_response(prompt)
```

这创建了一个反馈循环：使用工具的智能体生成失败数据，智能体然后使用这些数据改进工具描述，从而减少未来的失败。

### 测试工具设计

根据标准评估工具设计：明确性、完整性、可恢复性、效率和一致性。通过呈现代表性的智能体请求并评估结果工具调用来测试工具。

## 实用指导

### 避免的反模式

模糊描述："搜索数据库中的客户信息"留下太多未回答的问题。

神秘的参数名称：命名为 x、val 或 param1 的参数迫使智能体猜测含义。

缺失错误处理：以通用错误失败的工具不提供恢复指导。

不一致命名：在某些工具中使用 id，在其他工具中使用 identifier，在另一些中使用 customer_id 会造成混淆。

### 工具选择框架

设计工具集合时：
1. 识别智能体必须完成的不同的工作流
2. 将相关操作分组到综合工具中
3. 确保每个工具都有清晰、明确的目的
4. 记录错误案例和恢复路径
5. 用实际智能体交互进行测试

## 示例

**示例 1：设计良好的工具**
```python
def get_customer(customer_id: str, format: str = "concise"):
    """
    通过 ID 检索客户信息。
    
    在以下情况下使用：
    - 用户询问特定客户详情
    - 需要客户上下文进行决策
    - 验证客户身份
    
    参数：
        customer_id: 格式 "CUST-######"（例如，"CUST-000001"）
        format: "concise" 表示关键字段，"detailed" 表示完整记录
    
    返回：
        具有请求字段的客户对象
    
    错误：
        NOT_FOUND: 未找到客户 ID
        INVALID_FORMAT: ID 必须匹配 CUST-###### 模式
    """
```

**示例 2：糟糕的工具设计**

此示例演示了几个工具设计反模式：

```python
def search(query):
    """搜索数据库。"""
    pass
```

**此设计的问题：**

1. **模糊名称**："search" 是模糊的——搜索什么，用于什么目的？
2. **缺失参数**：什么数据库？查询应该采用什么格式？
3. **无返回描述**：此函数返回什么？列表？字符串？错误处理？
4. **无使用上下文**：智能体何时应该使用此工具与其他工具？
5. **无错误处理**：如果数据库不可用会发生什么？

**失败模式：**
- 智能体可能在应该使用更具体工具时调用此工具
- 智能体无法确定正确的查询格式
- 智能体无法解释结果
- 智能体无法从失败中恢复

## 指导原则

1. 编写回答做什么、何时做和返回什么的描述
2. 使用整合来减少模糊性
3. 实现响应格式选项以提高令牌效率
4. 为智能体恢复设计错误消息
5. 建立并遵循一致的命名规范
6. 限制工具数量并使用命名空间进行组织
7. 用实际智能体交互测试工具设计
8. 根据观察到的失败模式进行迭代
9. 质疑每个工具是启用还是限制模型
10. 优先选择原始的通用工具而非专用包装器
11. 投资于文档质量而非工具复杂性
12. 构建可以从模型改进中受益的最小架构

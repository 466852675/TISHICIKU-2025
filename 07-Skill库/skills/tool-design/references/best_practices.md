# 工具设计最佳实践

本文档为代理系统设计工具提供额外的最佳实践和指南。

## 工具哲学

工具是代理与世界之间的主要接口。与为理解底层系统的开发者设计的传统 API 不同，工具必须为由描述推断意图并从自然语言请求生成调用的语言模型设计。这一根本差异要求我们重新思考如何设计和记录工具接口。

目标是创建代理无需大量试错就能发现、理解和正确使用的工具。工具定义中的每个模糊性都成为潜在的失败模式。每个不清晰的参数名称都迫使代理猜测。每个缺失的示例都让代理在边缘情况下没有指导。

## 描述工程原则

### 原则 1：回答基本问题

每个工具描述应清楚回答四个问题。工具做什么？用具体术语准确说明工具完成什么，避免模糊语言如"帮助"或"可用于"。何时应该使用它？提供具体触发器和上下文，包括直接触发器和指示工具适用性的间接信号。它接受什么输入？用类型、约束和默认值记录参数，解释每个参数控制什么。它返回什么？描述输出格式和结构，包括成功响应和错误条件的示例。

### 原则 2：使用一致的结构

在整个代码库中保持工具描述的一致结构。当代理遇到新工具时，它们应该能够根据从其他工具学习的模式预测在哪里找到特定信息。这减少了认知开销，并防止因格式不一致导致的错误。

推荐结构包括第一句的简要描述、带有使用上下文的详细解释、具有清晰类型信息的参数部分、描述输出结构的返回部分，以及列出可能失败模式和恢复指导的错误部分。

### 原则 3：包含具体示例

示例弥合抽象描述和实际使用之间的差距。包括典型调用的示例，展示常见参数组合，边缘情况示例及如何处理它们，以及错误响应示例和适当的恢复操作。

好的示例是具体的而非通用的。不要用"使用像 '123' 这样的 ID"，而是用"使用格式：'CUST-######'（例如，'CUST-000001'）"。不要用"提供日期"，而是用"格式：'YYYY-MM-DD'（例如，'2024-01-15'）"。

## 命名约定

### 参数命名

参数名称应该是自文档化的。使用无需额外解释就能清楚表明目的的名称。除了广泛理解的缩写如 "id" 或 "url" 外，优先使用完整单词而非缩写。对相似概念使用一致的命名。

好的参数名称包括 customer_id、search_query、output_format、max_results 和 include_details。差的参数名称包括 x、val、param1 和 info。

### 枚举值

当参数接受枚举值时，在所有工具中使用一致的命名。对于布尔样式选项，对肯定选项使用前缀模式如 "include_"（include_history、include_metadata），对否定选项使用 "exclude_"（exclude_archived、exclude_inactive）。对于分类值，使用一致的术语，如 "format": "concise" | "detailed"，而不是在某些工具中混合使用 "format": "short" | "long" 而在其他工具中使用 "format": "brief" | "complete"。

## 错误消息设计

### 双重受众

错误消息服务于两个有不同需求的受众。调试问题的开发者需要详细的技术信息，包括堆栈跟踪和内部状态。从失败中恢复的代理需要可操作的指导，告诉它们出了什么问题以及如何纠正。

设计错误消息时，将代理恢复作为主要考虑因素。包括用清晰语言说明具体出了什么问题。提供描述代理下一步应该做什么的解决指导。为输入错误包含更正格式。添加有效输入示例。

### 错误消息结构

```json
{
    "error": {
        "code": "INVALID_CUSTOMER_ID",
        "category": "validation",
        "message": "Customer ID 'CUST-123' does not match required format",
        "expected_format": {
            "description": "Customer ID must be 9 characters",
            "pattern": "CUST-######",
            "example": "CUST-000001"
        },
        "resolution": "Provide a customer ID matching pattern CUST-######",
        "retryable": true
    }
}
```

### 常见错误模式

验证错误应指定收到什么、期望什么格式以及如何纠正。速率限制错误应指定等待时间和重试指导。未找到错误应建议替代方法或验证步骤。系统错误应指示是否适合重试并建议替代方案。

## 响应格式优化

### Token-准确性权衡

详细响应提供全面信息但消耗大量上下文 token。简洁响应最小化 token 使用但可能缺乏必要细节。最佳方法提供格式选项，允许代理为其需求请求适当的详细程度。

### 格式选项模式

```python
def get_customer_response(format: str = "concise"):
    """
    Retrieve customer information.
    
    Args:
        format: Response format - 'concise' for key fields only,
                'detailed' for complete customer record
    """
    if format == "concise":
        return {
            "id": customer.id,
            "name": customer.name,
            "status": customer.status
        }
    else:  # detailed
        return {
            "id": customer.id,
            "name": customer.name,
            "email": customer.email,
            "phone": customer.phone,
            "address": customer.address,
            "status": customer.status,
            "created_at": customer.created_at,
            "history": customer.history,
            "preferences": customer.preferences
        }
```

### 何时使用每种格式

对快速验证或简单查找使用简洁格式，当只需要确认时，以及在初始检索后的后续工具调用中。在基于客户数据做决策时使用详细格式，当输出成为其他处理的输入时，以及当完整上下文对正确性必要时。

## 工具集合设计

### 管理工具激增

随着代理系统增长，工具集合趋于激增。更多工具可以启用更多功能，但会造成选择挑战。研究表明工具描述重叠会导致模型混淆。关键洞察是，如果人类工程师不能明确说在给定情况下应该使用哪个工具，就不能期望代理做得更好。

### 合并指南

将代表单个工作流中顺序步骤的工具合并为处理整个工作流的单个工具。例如，不要分别使用 list_users、list_events 和 create_event，而是实现 schedule_event，在一个调用中找到可用性并安排。

即使它们共享某些功能，也要保持具有根本不同行为的工具分开。在不同上下文中使用的工具应保持分离以防止混淆。

即使在相似领域操作的工具之间也要保持清晰的边界。应通过仔细设计最小化重叠功能。

### 工具选择指导

在设计工具集合时，考虑代理需要哪些信息来做出正确选择。如果多个工具可应用于某种情况，在描述中澄清区别。使用命名空间创建逻辑分组，帮助代理导航工具空间。

## 测试工具设计

### 评估标准

根据清晰度、完整性、可恢复性、效率和一致性标准评估工具设计。清晰度衡量代理能否确定何时使用工具。完整性衡量描述是否包含所有必要信息。可恢复性衡量代理能否从错误中恢复。效率衡量工具是否支持适当的响应格式。一致性衡量工具是否遵循命名和模式约定。

### 代理测试模式

通过呈现代表性代理请求并评估结果工具调用来测试工具：

1. 准备具有多样代理请求的测试用例
2. 让代理为每个请求制定工具调用
3. 根据预期模式评估调用正确性
4. 识别常见失败模式
5. 根据发现改进工具定义

## 避免的反模式

### 模糊描述

差："搜索数据库中的客户信息。"这留下太多未回答的问题。什么数据库？什么信息可用？查询应该采用什么格式？

好："通过 ID 或电子邮件检索客户信息。当用户询问特定客户详细信息、历史记录或状态时使用。返回带有 id、name、email、account_status 和可选订单历史的客户对象。"

### 神秘参数名称

差：名为 x、val 或 param1 的参数迫使代理猜测含义。

好：名为 customer_id、max_results 或 include_history 的参数是自文档化的。

### 缺失错误处理

差：以通用错误或没有错误处理失败的工具。

好：提供特定错误类型、消息和解决指导的工具。

### 不一致命名

差：在某些工具中使用 id，在其他工具中使用 identifier，对某些概念使用 customer_id 而对其他概念使用 user_id。

好：对所有工具中相似概念保持一致的命名模式。

## 工具设计检查清单

在部署新工具之前，验证描述是否清楚说明工具做什么以及何时使用它。验证所有参数是否具有描述性名称和清晰的类型信息。验证返回值是否已记录结构和示例。验证错误情况是否已涵盖可操作消息。验证工具是否遵循其他地方使用的命名约定。验证示例是否展示常见使用模式。验证如果响应大小变化显著，格式选项是否可用。
